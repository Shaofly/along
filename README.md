# 圆个圈（Along）

一个只允许熟人进入、以真实关系和共同回忆为核心的私人社交平台。

它不试图替代微信、QQ 等即时通讯工具，也不追求公开传播、粉丝数量或内容热度。平台希望解决的是另一件事：让朋友们把已经发生的日常、共同经历和想留给未来的话，放进一个有边界、可长期保存的空间。

> 即时通讯负责“我们现在要做什么”，圆个圈负责“我们一起经历过什么”。

## 当前状态

当前版本：`0.1.0`。项目正在从视觉原型进入第一版最小可用开发。

- 已完成一版暖色调网页原型。
- 已接入 PostgreSQL、数据库迁移和持久化会话。
- 已实现邮箱密码登录、受保护主页和退出登录。
- 已实现首次部署时创建一位管理员和一位普通创始成员，两人自动成为朋友。
- 已实现共同邀请、邀请确认、一次性邀请码和邀请后自动成为朋友。
- 邀请码必须绑定注册邮箱，绕过邀请的直接注册会被服务器拒绝。
- 小圈子、动态持久化、图片上传和圈内协作管理尚未实现。
- 仓库只保留标准 Node.js 自托管方案，后续可部署到自租服务器并绑定自己的域名。

## 版本规划

项目采用语义化版本号 `主版本.功能版本.修订版本`：

- `0.x.x` 表示仍在逐步开发，数据结构和界面可能继续调整。
- 功能版本号用于标记一组可完整体验的新能力，例如从 `0.1.0` 升级到 `0.2.0`。
- 修订版本号用于兼容性修复、文档补充和不改变核心功能的小调整，例如 `0.1.1`。
- `1.0.0` 表示核心产品稳定、部署与备份流程完整，适合在正式服务器长期运行。

## 技术架构

- Next.js 16 + React 19 + TypeScript。
- PostgreSQL 17。
- Drizzle ORM 与 SQL 迁移文件。
- Better Auth 管理密码哈希、数据库会话和安全 Cookie。
- 标准 Node.js 自托管运行方式，生产构建支持 standalone 输出。
- 后续使用对象存储保存照片、视频和语音，不把用户媒体长期堆在应用服务器磁盘中。
- 正式版本面向自租服务器上的标准 Next.js、Node.js、PostgreSQL 和对象存储部署。

## 产品定位

圆个圈（Along）是一个“熟人才进得来”的共同生活档案，而不是缩小版的公开社交平台。

平台关注：

- 真实关系，而不是关注与粉丝关系。
- 有限可见，而不是默认全员公开。
- 共同保存，而不是追求即时热度。
- 生活本身，而不是精心包装的人设。
- 长期回忆，而不是快速消失的信息流。

平台不鼓励：

- 公开广场和陌生人推荐。
- 转发、热搜、榜单和流量排名。
- 粉丝数、获赞排名等社交压力指标。
- 即时聊天、群聊和复杂日程安排。
- 实时定位或持续追踪朋友位置。

## 产品与 UI/UX 规范

本节记录已经讨论并确认的产品体验结论。后续页面设计和功能开发应以这些规范为准；尚未确认的方案不作为实现依据。

### 品牌名称

- 中文名称暂定为“圆个圈”。
- 英文名称暂定为“Along”。
- “圆个圈”表达熟人围成一个有边界的小圈，也有把共同经历圈起来、留存下来的含义。
- “Along”不作为中文名称的逐字翻译，而是表达一路同行、在彼此身边的关系感。

### 产品气质

整体风格采用“现代实用为主，手账温度为辅”的方向：安静、真诚、有生活感，像一本由朋友共同维护的生活册，同时保持现代产品应有的清晰和易用。

- 温暖但不煽情，亲密但不黏腻。
- 有手账和书刊气息，但不做成儿童绘本，也不牺牲日常使用效率。
- 不追求热闹和注意力竞争，允许页面保留安静、舒展的空间。
- 避免“精彩人生”“闪闪发亮”等刻意抒情或宣传式文案。
- 日常界面需要清楚易用，但不能过于冷淡或极简；温馨感应贯穿整个产品，而不只出现在回忆页面。
- 回忆照片、时间胶囊等展示可以比日常动态更有书刊感和仪式感。

### 视觉方向

- 保留暖白、墨绿作为基础色，并以珊瑚粉、柔黄、灰蓝和浅绿色区分内容与状态。
- 减少大面积同色背景和装饰性渐变，温馨感主要来自照片、色彩细节、排版和真实文案。
- 日常页面采用克制的小标题；超大标题只用于欢迎页、回忆专题等少数场景。
- 正文使用清晰的中文系统字体，手写或书刊感字体只用于少量回忆标题，不承担正文阅读。
- 真实照片是最重要的视觉内容。照片应尽量保留自然比例，不为版式随意裁掉人物。
- 拍立得等手账效果集中用于回忆模块，不扩散到所有功能页面。
- 朋友、圈子等真正独立的重复项目可以使用 6 至 8 像素圆角卡片；动态信息流默认采用无外框的连续内容布局，不把每条动态放进与页面背景明显分离的白色容器。
- 动效应短暂、自然并用于解释状态变化，例如照片浮到前层、弹窗进入和发布完成反馈。
- 界面应尊重系统的“减少动态效果”设置。

### 文案方向

界面文案应像朋友之间自然、明确地说话，不使用广告口吻或刻意制造感动的句子。例如：

- “最近动态”
- “七月一起留下的事情”
- “阿澈补充了两张照片”
- “仅小圈子成员可见”
- “这段记录创建于 2026 年 7 月 15 日”

### 信息架构与全局导航

登录后的首页是综合入口，“回忆”作为首页中的内容表达，不再设置一个笼统的独立一级回忆页。主要页面关系如下：

- **首页**：问候、发布入口、随机照片与暖心小纸条、条件出现的回忆内容、最新 5 条动态和常用功能入口。
- **动态**：从首页“查看全部动态”进入的完整信息流。
- **圈子**：活跃圈子、历史圈子和圈子审批。
- **朋友**：朋友列表、朋友主页和共同邀请。
- **足迹**与**胶囊**：从首页固定功能入口进入的特色功能页，不长期占据桌面主导航。

通知使用铃铛图标进入；个人空间、设置和仅管理员可见的管理入口放在头像菜单中；搜索使用顶栏图标进入独立页面，不占一级导航。

#### 电脑与平板

- 顶部使用固定但不厚重的导航栏。
- 左侧显示“圆个圈 Along”品牌标志，中间显示“动态、圈子、朋友”，右侧显示搜索、通知和头像。
- 当前栏目必须有明确的选中状态。
- 点击品牌标志始终返回首页。
- 足迹和胶囊在首页辅助区域提供稳定入口，头像菜单保留完整导航作为备用路径。

#### 手机

- 暂不设置底部导航。
- 一级页面顶部使用“左侧头像、中间圆个圈品牌、右侧搜索与通知”的紧凑布局。
- 进入内容详情等二级页面后，左侧头像替换为返回按钮，并根据空间精简中间品牌。
- 首页开头固定提供“圈子、朋友、足迹、胶囊”四个图标入口。
- 头像菜单保留完整导航，作为任何页面都能使用的备用入口。
- 进入具体内容后显示返回按钮，并允许用户沿作者、圈子、记录和朋友等内容关系自然跳转。
- 发布入口属于首页和对应圈子，不设置持续遮挡内容的全站悬浮按钮。

#### 页面层级

```text
首页
├── 发布
├── 随机照片与暖心小纸条
├── 条件出现的回忆与胶囊状态
└── 最近五条动态

动态
├── 完整动态流
├── 动态详情
└── 作者主页

圈子
├── 活跃圈子
├── 历史圈子
├── 圈子详情
├── 成员与审批
└── 圈内记录详情

朋友
├── 朋友列表
├── 朋友主页
├── 共同经历
└── 邀请与确认

足迹
├── 游玩记录
└── 朋友地图

胶囊
├── 写给自己或朋友
├── 小圈子胶囊
```

#### 返回与链接行为

- 浏览器、系统手势和页面内返回按钮都必须正常工作。
- 页面内返回按钮返回真实上一层，不固定跳回首页。
- 从通知进入内容后，返回时应回到通知列表；从朋友主页进入动态后，返回时应回到朋友主页。
- 刷新页面后仍停留在当前内容，不能因为刷新丢失页面位置或上下文。
- 登录用户打开站内深层链接时，应在权限允许的情况下直接进入对应内容。
- 无权查看时只显示明确的无权限说明，不泄露内容标题、参与者或其他敏感信息。

以上结构是当前设计基线，并非不可变更。后续可以依据用户测试调整，但调整时应同步更新文档、页面和交互规则。

### 登录后首页

登录后的首页采用“轻量总览在前、最近动态在后”的布局。现有视觉原型中的大型宣传标题不继续作为登录后的日常首页主体，可以在后续改造为登录前欢迎页或首次进入时的欢迎内容。

#### 页面顶部

- 保留紧凑的品牌栏，显示“圆个圈 Along”、通知和头像。
- 保留一行简短、自然的时间问候，例如“晚上好，小林”，不附加刻意抒情的宣传句。
- 手机首页提供圈子、朋友、足迹和胶囊四个图标入口。
- 常见手机首屏内至少露出发布入口和一项真实内容信号，不能让问候或装饰占满整个首屏。

#### 最近动态

- 首页只显示最新 5 条当前用户有权查看的朋友个人动态、圈子共同记录和自己的内容，不在首页无限加载。
- 首页 5 条动态复用完整动态组件，正文可以原地展开，并按统一规则最多显示 10 条评论。
- 当前阶段严格按照发布时间排列，不做算法推荐、热度排序或付费曝光。
- 动态卡片应清楚显示作者、头像、发布时间、发布空间、可见范围和最后编辑状态。
- 不因自动刷新突然移动用户正在阅读的内容；返回首页时应尽量恢复之前的滚动位置。
- 最近动态末尾提供“查看全部动态”，进入独立动态页面后才显示“全部 / 朋友 / 圈子”筛选并使用游标持续加载。

#### 发布入口

- “最近动态”标题附近提供明确的“记录一下”入口。
- 所有内容复用同一个发布器，在发布器顶部选择“日常动态”或“游玩记录”。
- 发布个人内容还是圈子内容由发布空间选择决定，不再把“个人近况”和“圈内动态”拆成两个发布类型。
- 圈子详情页同时保留就近入口，打开后自动预选当前圈子，但用户仍可返回修改。
- 发布入口不使用持续遮挡内容的悬浮按钮，也不需要先弹出多层类型菜单。

#### 回忆照片与暖心小纸条

首页保留视觉原型中照片与文字框层叠的设计，但它们必须成为真实、可操作的内容，而不是静态装饰。

- 照片从当前用户自己的个人动态，以及当前仍有权查看的圈子动态或游玩记录中受控随机选择；不抽取朋友仅发布在其个人空间中的照片。
- 删除、隐藏、归档、来源失效或当前用户失去查看权限的照片必须立即退出候选池。
- 默认一次选择少量且互不重复的照片；照片不足时自动减少层数，没有合适照片时不显示空占位。
- 随机结果在一段时间内保持稳定，默认按周轮换，并避免短期重复；可以提供一个低调的换一组图标供用户主动刷新。
- 电脑端可在动态流右侧显示由照片卡和文字卡组成的层叠区域。
- 点击某张照片或某张文字卡时，该卡片自然移动到最前层，并露出日期、来源或简短说明。
- 打开来源动态或照片使用独立、明确的操作，避免“选中卡片”和“打开详情”含义冲突。
- 手机端使用横向滑动的照片与文字卡组，不强行复刻桌面端的重叠操作。
- 文字卡暂称“暖心小纸条”，内容使用自然、克制而温暖的短句。
- 小纸条文案来自可维护的文案库，后续允许管理员添加、停用和调整，不应为了更换文案修改页面代码。
- 文案与照片一样采用受控随机并保持一段时间稳定，不调用大模型，也不在每次刷新时变化。
- “那年今天”、地图故事和时间胶囊只在确有内容时作为独立小区域出现，不穿插或伪装成普通动态；没有内容时整个区域隐藏。

#### 响应式布局

- 电脑端以动态流为主栏，回忆卡组和圈子近况作为辅助栏。
- 平板根据实际宽度采用双栏或单栏，不能仅按设备名称判断布局。
- 手机端采用单列结构，回忆卡组改为横向滑动。
- 不自动播放视频，不使用会干扰阅读的持续动画。

#### 首页入口与未读状态

- 首页“圈子”入口在任一圈子有未读内容或待处理事项时显示汇总圆点。
- 首页“朋友”入口在存在朋友个人动态未读、好友申请或共同邀请待办时显示汇总圆点。
- “胶囊”入口可以在胶囊新开启或需要当前用户完成提交时显示圆点；“足迹”不重复承接圈子新动态的未读状态。
- 首页入口只显示是否存在更新，不堆叠动态总数；进入对应页面后再显示具体项目。
- 未读提示使用柔和珊瑚色、保持静止且不闪烁；只有通知或审批确实需要数量时才显示数字。
- 未读阅读位置保存在服务端，并按当前用户实际可见内容计算，不能由前端拿全部内容后自行判断。

## 核心原则

### 关系有远近

同处一个平台并不代表彼此认识。用户默认只能向自己的朋友、小圈子或指定成员发布内容，非朋友之间默认互不可见。

### 加入需要共同信任

注册必须使用邀请码。邀请码不能由一个人随意生成，需要至少 2 位、最多 5 位真正认识新人的现有成员共同邀请。

### 个人表达与共同回忆分开

个人空间记录“我想表达什么”，小圈子记录“我们共同经历了什么”。二者在归属、可见范围和编辑权限上采用不同规则。

### 管理边界在发布时确定

小圈子内容由创建者在发布时选择“仅我管理”或“圈内共同管理”。管理方式决定谁可以修改和删除。创建者可以在发布后把“仅我管理”升级为“圈内共同管理”，但共同管理不能再降回仅自己管理。平台不保存面向用户的完整版本历史，只保留创建信息以及最后编辑者和最后编辑时间，让共同记录保持轻松，也让管理边界足够明确。

### 温暖不等于没有边界

所有内容都必须有清晰的可见范围。照片地点、参与者、圈子成员和时间胶囊等敏感信息不能因为关联或转发突破原始权限。

## 用户与关系模型

### 平台成员

通过有效邀请码完成注册的人。成为平台成员不等于与所有其他成员成为朋友。

### 管理员

首次设置时创建的第一位成员自动成为管理员。管理员是平台维护者，不是可以绕过隐私权限的超级用户。

- 管理成员状态和处理账号异常。
- 处理违法违规、隐私泄露或人身安全相关的紧急隐藏。
- 查看系统运行状态和审计记录。
- 维护邀请、存储和服务器配置。
- 不能绕过朋友或小圈子的可见范围查看私人内容。
- 不能仅凭管理员身份编辑或删除私人内容；紧急情况只能按安全流程暂时隐藏。
- 不能代替朋友确认邀请或代替圈子成员进行表决。

### 朋友

双方存在直接朋友关系时，才能看到彼此面向朋友发布的内容。

- 新用户注册后，自动与共同邀请自己的成员成为朋友。
- 其他成员不会自动成为朋友。
- 个人内容默认对朋友可见，但发布者可以缩小可见范围。
- 同一小圈子中的非好友可以从有限个人主页发起好友申请，对方明确同意后才建立朋友关系。
- 被拒绝后不显示拒绝原因，并设置一段时间内不能重复申请，避免持续打扰。
- 不提供“可能认识的人”推荐，也不能通过全平台昵称搜索陌生成员。

现实中认识但没有共同圈子的已注册成员，可以使用短期好友链接、二维码或好友码建立联系：

1. 一方在朋友页面生成好友链接或好友码，并通过站外方式发送给认识的人。
2. 对方必须先登录圆个圈才能打开；好友链接不能用于注册平台。
3. 打开后只展示昵称、头像和简介等最小资料，对方提交好友申请。
4. 链接创建者再次确认后，双方正式成为朋友。

- 好友链接默认 7 天有效，使用一次后失效，创建者可以提前撤销。
- 服务端只保存好友令牌的哈希，不能把可使用的原始令牌直接存入数据库。
- 即使链接被转发，陌生人也只能提交申请，不能直接成为朋友或查看个人动态。
- 好友关系来源包括共同邀请、共同圈子申请和好友链接申请，系统保留来源与建立时间。
- 任一方可以解除朋友关系，不向对方发送“被删除”通知。
- 解除后立即失去基于朋友关系获得的内容权限，但双方共同圈子的内容和历史权限不受影响。

### 小圈子

由几位朋友共同建立的私密空间，例如寝室朋友、旅行伙伴或长期好友组。

- 圈内成员权利平等，不设置拥有绝对权限的“圈主”。
- 创建圈子时发出的初始邀请只在 3 天内有效，过期后不能突然接受并自动加入已经运行的圈子。
- 新成员加入必须获得所有活跃成员同意。
- 成员可以退出圈子的活跃关系，退出不等于抹去成员期间的共同历史。
- 退出后不再看到新增内容，也不能继续发布、评论或管理圈内内容。
- 退出时形成历史权限快照，成员仍可只读查看自己在成员期间有权访问的记录。
- 退出成员上传的内容和历史参与记录继续保留，不会自动删除或匿名化。
- 任何成员都可以在自己的界面中封存或隐藏圈子，但不会影响其他成员。
- 小圈子解散需要所有活跃成员同意。
- 解散后内容转为只读共同档案，而不是被删除。

### 小圈子页面

小圈子列表区分“活跃圈子”和“历史圈子”。圈子条目可以显示封面、名称、成员头像、最近更新时间和当前用户的未读状态，不展示圈子活跃度排名。

- 圈子页面自动列出当前用户的全部活跃圈子、历史圈子和待处理审批，不要求用户手动添加到栏目。
- 别人发布了当前用户有权查看的新圈子内容，或圈内产生新的成员关系等系统动态时，对应圈子显示未读圆点。
- 当前用户自己发布的内容不为自己产生未读；评论和回应主要进入通知中心，不重复点亮整个圈子。
- 用户进入圈子并成功加载可见的最新内容后，服务端更新该用户的圈子阅读位置；手机和电脑共享同一未读状态。
- 重要圈子仍同时遵循重要订阅通知规则，未读圆点与站内通知互不替代。

#### 手机端圈子动态页

点击圈子后默认进入连续动态流，而不是先进入功能导航页。页面使用类似`/circles/:id`的独立路由。

- 顶部保留紧凑的圈子名称、返回和更多操作，不使用“最近 / 游玩 / 相册”吸顶标签栏。
- 顶部提供足迹、相册和成员等少量内容入口，随后直接进入“记录一下”和持续加载的圈子动态。
- 成员入口可以使用少量重叠头像和“4 位成员”等人数信息，点击后直接进入成员页，不藏在右上角更多菜单中。
- 足迹入口同时承载游玩记录与地图浏览；进入后通过页面底部的“记录 / 地图”连体分段按钮切换，不在圈子主页拆成两个入口。
- 顶部内容入口随页面滚动自然离开视野，不长期占据动态阅读空间。
- 动态流使用基于游标的分页加载，并始终执行圈子成员周期和历史权限校验。

#### 圈子扉页

圈子封面不是动态页中简单拉高的折叠区域，而是使用类似`/circles/:id/overview`的独立页面路由。它像共同生活册的扉页，手机端控制在一个完整视口内，不在扉页内部继续堆叠长列表。

- 动态页顶部露出一小段可识别的封面区域和展开提示。
- 用户从封面区域向下拖动时先看到扉页预览；超过触发距离并松手后完成页面切换，地址栏同步变为扉页路由。
- 未达到触发距离时自然回弹，不改变地址；页面其他区域的下拉仍保留浏览器默认行为，避免与下拉刷新冲突。
- 点击封面区域或展开提示也能直接进入扉页，手势不能成为唯一入口。
- 扉页展示圈子主照片或克制拼贴、名称、简介、成员、随机共同回忆，以及足迹、相册和时间胶囊的少量入口。
- 回忆文字直接使用来源动态的原文片段，按确定的字符长度截断，不调用大模型生成摘要或改写朋友的表达。
- 来源动态没有正文时，只显示日期、允许展示的地点、上传者和来源记录，不凭空生成温暖文案。
- 封面可以由成员指定固定照片，也可以从当前用户有权查看的圈内历史中按稳定规则选择回忆照片和原文片段；不得为了构图裁掉重要人物。
- 圈子扉页只负责展示，不提供编辑名称、简介、封面、成员或其他圈子资料的入口。
- 修改圈子资料从圈子动态页右上角的更多菜单进入“圈子设置”；活跃成员完成修改后生成对应圈子系统动态。
- 不展示活跃排名、成员贡献排名或制造比较压力的累计数字。

#### 从扉页返回动态

- 手机端在扉页向上滑动，超过触发距离并松手后返回圈子动态路由。
- 扉页本身限制为一个视口，避免“向上滚动扉页内容”和“上滑返回”产生含义冲突。
- 底部安全区域提供带向上箭头和“回到动态”文字的操作按钮，使用圆角不超过 8 像素的细白色边框，并确保在任何封面照片上都有足够对比度。
- 点击按钮、使用浏览器返回或上滑手势都进入同一个返回流程；直接打开扉页链接时，返回操作应进入对应圈子动态，而不是把用户带到不可预测的外部页面。
- 返回后恢复动态页原来的筛选状态和滚动位置。
- 手势动画只负责解释页面方向，路由和返回功能不能依赖动画；系统开启“减少动态效果”时使用简化过渡。

#### 平板与桌面端

- 平板竖屏沿用手机的单列动态流；横屏在空间足够时逐步展开辅助区域。
- 桌面端以中间的连续动态流为主，左侧或右侧展示圈子资料、足迹、相册预览和时间胶囊入口。
- 侧栏使用与页面连续的背景和清楚分区，不堆叠装饰性悬浮卡片。
- 桌面端仍可进入独立圈子扉页，但以点击封面入口为主，不强行模拟手机上下拉手势。

#### 圈子成员页

- 从圈子动态页顶部的成员入口、圈子扉页成员头像或桌面侧栏进入同一个成员页。
- 页面区分“当前成员”和“曾经的成员”；同一个人多次加入和退出时只显示一个成员条目，不重复制造多个身份。
- 成员详情可以简洁展示多段成员周期，例如“2026 年 1 月加入，3 月退出，5 月重新加入”。
- 当前成员条目显示头像、昵称、与当前用户的朋友关系以及模糊的“最近来过”状态。
- 点击成员进入其有限个人主页；同圈但尚未成为朋友时可以发起好友申请。
- 待处理的新增成员和重新加入申请显示在成员页顶部，处理完成后自动离开待办区域。
- 不显示圈主、管理员、成员贡献量或活跃排名；圈子资料和成员关系操作仍从圈子动态页右上角的圈子设置进入。

“最近来过”反映最近一次已认证活动，而不是可能数月不变的登录时间，也不提供实时在线状态。显示档位暂定为：

- 今天来过。
- 近 7 天来过。
- 近 30 天来过。
- 很久没来。
- `-`，表示该成员选择隐藏。

- 默认显示模糊状态，成员可以在个人隐私设置中关闭。
- 只有直接朋友和处于同一活跃圈子的当前成员可以查看；已退出成员不能继续查看其他成员退出之后的活跃状态。
- 邀请页、搜索结果和没有当前朋友或活跃圈子关系的有限个人主页不展示该状态。
- 服务端最多每 15 分钟更新一次`last_seen_at`，避免每次页面请求都写数据库。
- 权限层只向前端返回计算后的状态档位，不返回精确活动时间；隐藏后统一返回不可见状态并显示`-`。

#### 圈子系统动态

圈子名称、简介、封面和成员关系等变化以“圈子系统动态”显示在圈子内部，例如“阿澈修改了圈子简介”。

- 任意活跃成员都可以修改圈子名称、简介和封面。
- 每次修改记录操作者、时间以及必要的变更内容。
- 系统动态由系统生成，不属于普通生活动态，也不使用普通内容的管理方式。
- 系统动态只出现在圈子内部，不进入任何成员的个人主页、共同经历或回忆时间线。
- 系统动态不能被成员当作普通内容编辑或删除，但后续可以在界面中折叠显示，避免打断内容浏览。

## 邀请与注册

### 目标流程

1. 一位现有成员发起邀请提案。
2. 发起者邀请另外 1 至 4 位成员共同确认自己认识该新人并愿意邀请其加入。
3. 已确认的邀请人达到 2 位后，不自动生成邀请码；发起者可以继续等待其他人确认，或主动点击“生成邀请码”。
4. 生成时锁定最终确认名单，最多为 5 位；尚未确认的人不再计入本次邀请。
5. 新人使用邀请码注册。
6. 注册完成后，新人与最终锁定名单中的所有共同邀请人自动成为朋友。
7. 系统保留邀请来源，例如“由小林、阿澈和眠眠共同邀请加入”。

### 邀请码要求

- 一次性使用。
- 设置有效期限。
- 每个邀请码必须绑定至少 2 位、最多 5 位已确认的邀请人。
- 达到最低人数后由发起者决定何时结束确认并生成邀请码，生成后邀请人名单不可继续增减。
- 与邀请提案绑定，不能转给其他人使用。
- 使用后立即失效。
- 邀请发起、确认、取消和使用均留下记录。
- 不设置“只信任发起者但不认识新人”的代担保通道；每位最终邀请人都必须认识新人，并愿意在注册后与其建立朋友关系。

### 邀请注册页面

- 平台不提供公开注册入口，只提供登录、使用邀请加入和首次部署初始化。
- 邀请页面显示共同邀请人的昵称、邀请剩余有效期和必要说明，不显示邀请人的邮箱。
- 注册邮箱必须与邀请码绑定邮箱一致。
- 头像可以跳过，未设置时使用昵称首字和柔和主题色。
- 注册成功后自动登录，避免让新人重复输入密码。
- 首次进入只展示简短欢迎内容、共同邀请人和基本权限说明，不使用连续多页教程或大量教学气泡。
- 普通注册邀请暂按 7 天有效期设计；建圈初始邀请单独使用已经确定的 3 天有效期。

注册按钮前必须以显著、清晰但不过度打扰的方式展示数据处理摘要，并提供《用户协议》和《隐私政策》完整入口。确认框默认不勾选，用户主动同意后才能注册。

数据处理摘要必须诚实说明：

- 圆个圈根据朋友、小圈子和内容权限展示信息，不向无权限用户提供内容。
- 为支持搜索、圈内协作编辑和回忆整理，动态与评论正文不做端到端加密或应用层密文存储。
- 应用服务器在提供功能时需要读取正文，但管理员后台不提供浏览私人正文的日常入口。
- 照片和附件计划保存在私有对象存储中，用户通过内容权限校验后获得短期访问链接。
- 精确位置属于单独的可选能力，不捆绑在注册同意中；第一次使用当前位置或精确地图地点时另行说明和确认。

### 数据安全基线

- 当前 Better Auth 默认使用 `scrypt` 对密码进行不可逆处理，数据库不保存原始密码，管理员不能查看或还原用户密码。
- 暂不为更换算法而自行改造认证核心；后续如切换 Argon2id，需要设计旧密码哈希的兼容迁移。
- 数据库迁移账号、应用运行账号、备份账号和人工运维账号应分离，并遵循最小权限原则。
- 应用运行账号必须具备提供搜索和内容展示所需的读取能力，不能虚假承诺“数据库账号完全无法读取正文”。
- 所有内容访问必须通过统一服务端权限层；前端隐藏、管理员身份或客户端筛选都不能代替权限校验。
- 正式部署启用 HTTPS、加密云硬盘、加密备份、私有对象存储和对象存储服务端加密。
- 日志禁止记录密码、完整动态正文、长期媒体签名链接和不必要的精确位置。
- 普通管理员不拥有私人内容浏览入口。紧急访问机制只有在至少存在两位独立安全审批人并完成技术实现后，才能对用户承诺共同审批、限时授权和可见审计记录。

第一版不单独建设隐私工单或自助账号注销系统。暂未提供的查阅、处理或隐私能力统一通过“反馈”入口提交，并提供“隐私与数据”分类；后续业务完善时再增加数据导出、删除、注销和处理进度等正式流程。

## 内容归属模型

每条内容需要区分以下概念：

- **创建者**：最初发起内容的人。
- **发布空间**：个人空间或某个小圈子。
- **参与者**：真正参与该事件的人，可以少于圈子总人数。
- **可见范围**：哪些人有权查看。
- **管理方式**：仅创建者管理，或由圈内活跃成员共同管理。
- **最后编辑者**：最近一次修改正式内容的人。

这几个概念彼此独立。例如，一个四人小圈子的聚餐记录可以只标记三位参与者，但仍发布到四人小圈子并由四人查看。被标记为活动参与者、发表评论、查看内容或作出回应都不会改变内容的管理方式。

## 发布模式

发布时不使用含义模糊的“允许他人修改”复选框，而是先明确选择发布空间：

```text
发布到：  [ 我的空间 ]  [ 小圈子 ]
```

### 统一发布器

首页和圈子中的“记录一下”入口进入同一个移动端优先的发布器。页面顶部使用分段选择切换内容类型：

```text
[ 日常动态 ]  [ 游玩记录 ]
```

发布器采用一张自然向下滚动的编辑页面，不把日常发布拆成多步向导。底部可以固定发布操作栏，但它只服务当前编辑任务，不属于全站底部导航。

#### 日常动态编辑

- 输入正文并添加最多 20 张照片。
- 选择发布到“我的空间”或某个小圈子。
- 发布到个人空间时设置可见范围。
- 发布到小圈子时选择`仅我管理`或`圈内共同管理`，默认`仅我管理`。
- 心情和相关朋友均为可选项，不让轻量发布变成填表。

#### 游玩记录编辑

- 选择实际发生日期和活动类型，例如聚餐、旅行、散步、电影、探店或自定义类型。
- 必须选择一个小圈子，并从该圈子成员中选择实际参与者；参与者可以少于圈子总人数。
- 可以设置地点、地图展示级别、正文和最多 20 张照片。
- “加入朋友地图”默认勾选，用户可以在发布前关闭。
- 管理方式默认`圈内共同管理`。

#### 发布交互

- 普通动态只显示必要字段，选择游玩记录后才展开日期、参与者和地点等结构化字段。
- 发布按钮附近低调显示当前设置摘要，例如“周末小队 · 圈内可见 · 共同管理”，点击摘要可返回对应设置。
- 不在普通发布前展示大段权限说明或强制二次确认。
- 草稿为空时不显示草稿提示；存在草稿时才显示“还有 N 条未完成的记录”。
- 每张图片独立显示上传进度和失败重试，未完成上传时不能静默发布缺图内容。
- 发布请求必须使用防重复标识，避免网络迟延或连续点击生成重复内容。
- 从共同相册直接上传时复用同一个发布器，自动预选当前圈子、日常图片动态和`圈内共同管理`，发布后同时进入圈子动态和相册。

### 发布到我的空间

- 内容属于个人表达。
- 默认对朋友可见。
- 可以设置为仅自己、指定朋友或指定范围可见。
- 只有发布者可以编辑正文。
- 其他用户可以在权限允许时评论或回应，但不能修改正文。
- 保留最初创建时间；修改后额外显示最后编辑时间。
- 发布者可以修改可见范围、归档或彻底删除自己的个人内容。
- 彻底删除后，正文、媒体和该内容下的评论与回应不再向任何用户展示。
- 不允许其他用户将内容转发到圈外。

### 发布到小圈子

- 必须选择一个小圈子。
- 内容默认仅该小圈子成员可见。
- 发布时必须选择管理方式：`仅我管理`或`圈内共同管理`。
- `仅我管理`时，只有创建者可以修改或删除整条内容，其他成员不能向该内容补充照片或资料。
- `圈内共同管理`时，任一圈内活跃成员都可以修改或删除整条内容，也可以补充文字、照片、地点和参与者等信息。
- 内容创建者可以在发布后把`仅我管理`切换为`圈内共同管理`。
- 切换是不可逆的权限升级；进入`圈内共同管理`后不能再改回`仅我管理`。
- 切换前进行一次简短确认，说明切换后圈内活跃成员都可以修改或删除该内容。
- 切换本身不属于正文编辑，不改变创建时间、最后编辑者或最后编辑时间，但应通知圈内成员并记录必要的权限变更审计元数据。
- 普通圈子动态默认选择`仅我管理`。
- 游玩记录默认选择`圈内共同管理`。
- 从共同相册直接上传而自动创建的图片动态默认选择`圈内共同管理`。
- 保留最初创建者和创建时间。
- 编辑不会改变创建时间。
- 页面显示最后编辑者和最后编辑时间。
- 不向用户提供完整版本列表、修改差异或旧版本恢复。
- 删除整条内容时，正文、媒体、评论、回复和回应一并删除；删除前明确提示影响范围。
- 退出圈子活跃关系的成员对历史内容只读，即使内容原本采用圈内共同管理，也不能再修改或删除。

示例：

```text
阿澈创建于 2026 年 7 月 15 日
眠眠最后编辑于 2026 年 7 月 16 日 21:34
```

### 正式发布与误操作

平台让管理规则保持轻量，但仍需处理误删、隐私泄露、违法内容和人身安全风险。

- 发布个人内容和小圈子内容都不强制弹出二次确认，避免让日常记录显得沉重。
- 发布按钮附近应以低调但清楚的方式显示发布空间、可见范围和管理方式，例如“周末小队 · 圈内可见 · 共同管理”。
- 用户可以主动打开预览或规则说明，但普通发布流程不被说明弹窗阻断。
- 删除操作真正发生时，再显示影响范围和确认，不在发布前反复强调删除规则。
- 删除权限由内容当前的管理方式决定，不根据是否已经有人编辑过动态变化；管理方式只允许从仅我管理升级为圈内共同管理。
- 隐私泄露、人身安全或违法违规内容可以进入紧急隐藏和管理员审核流程。
- 隐藏或处理操作应保留审计记录。
- 当前版本暂不设计删除账号功能。

### 草稿与自动保存

草稿采用“服务端为主、本地兜底”的方式保存。

- 正式草稿保存在服务端，因此用户可以在手机、平板和电脑之间继续编辑。
- 草稿只有作者本人可以读取，不会出现在动态流、圈子、个人空间、通知或搜索结果中。
- 浏览器本地保存短期恢复副本，用于断网、页面崩溃或意外关闭后的恢复。
- 服务端同步成功后，本地副本只作为短期容错缓存，不能成为唯一正式数据来源。
- 文字修改应在短暂停顿后自动保存，图片等较大内容显示独立上传进度。
- 编辑器只使用轻量状态提示，例如“保存中”“已保存”或“离线，稍后同步”，不频繁弹窗。
- 草稿阶段不产生正式创建信息或最后编辑信息；正式发布后才记录创建者和创建时间。
- 圈子草稿在发布前仍属于作者个人，其他圈内成员不能提前看到或参与编辑。

### 个人空间中的共同经历

用户参与的小圈子内容可以显示在其个人空间中，但这只是对原内容的聚合展示，不会复制内容、改变内容归属或扩大可见范围。

- 个人空间区分“个人动态”和“共同经历”，默认可以在同一条连续内容流中混合展示。
- 用户是创建者、活动参与者或媒体上传者时，相关圈子内容可以进入其“共同经历”。
- 圈子内容仍以原圈子中的记录为唯一正式版本，编辑和删除规则继续由原记录决定。
- 用户自己可以看到与自己相关且仍有历史访问权的内容。
- 其他人访问该用户的个人空间时，必须同时拥有原圈子内容的查看权限，才能看到对应的共同经历。
- 没有原始权限的人不会看到占位卡片、标题、照片、地点、参与者或圈子名称。
- 成员退出圈子的活跃关系后，成员期间有权查看且与自己相关的历史内容仍可以显示；退出之后新增的内容不会进入其个人空间。

### 个人主页与朋友主页

个人主页用于呈现一个人的表达和共同经历，不作为公开简历或社交数据展示页。平台不显示粉丝数、累计获赞数等会制造比较压力的指标。

#### 主页资料区

- 显示头像、昵称和一句简短个人介绍。
- 允许用户设置个人封面照片或主题色，但封面不能占满常见手机的整个首屏。
- 封面的具体比例、裁切安全区、推荐分辨率和不同设备适配规则在响应式视觉规范中统一确定。
- 编辑资料操作只对本人显示。
- 根据双方关系和个人隐私设置显示模糊的“最近来过”状态，不展示精确时间或实时在线。

#### 内容栏目

个人主页暂不单独设置“回忆”栏目，避免与共同经历的内容范围重复。资料区下方直接进入连续内容流，并提供轻量筛选：

- **全部**：混合展示当前访问者有权查看的个人动态和共同经历。
- **个人动态**：本人发布且当前访问者有权查看的个人内容。
- **共同经历**：与本人相关且当前访问者同时拥有原始权限的圈子内容。
- 本人查看自己的主页时，可以额外进入草稿、仅自己可见内容和个人归档；这些入口不对其他人展示。
- 内容继续使用与首页一致的连续背景和轻分隔，不因为进入个人主页改成明显卡片墙。

#### 主页进入路径

朋友主页不是只能从圈子成员页进入。所有入口都指向同一套个人主页路由，并在服务端按当前访问关系裁剪内容：

- 首页或全局菜单中的“朋友”入口进入自己的朋友列表，点击任一朋友进入其主页。
- 点击动态作者、评论者或回应者的头像和昵称进入其主页。
- 从相关站内通知点击发起人，可以进入其有权限展示的主页。
- 圈子成员页和圈子扉页的成员头像可以进入主页。
- 访问共同朋友资料时，点击双方共同朋友可以继续进入其主页。
- 人物搜索仍只允许搜索直接朋友、当前或历史共同圈子成员，不建立陌生人发现入口。

朋友列表只展示当前用户自己的直接朋友，可以在列表内按昵称搜索；不使用活跃度排名或“可能认识的人”推荐。没有圈子关系但通过共同邀请或好友链接建立的朋友，也始终可以从这里进入主页。

- 朋友页面自动列出全部直接朋友，不要求用户另行添加到栏目。
- 朋友发布当前用户有权查看的新个人动态时，其朋友条目可以显示未读圆点；该朋友发布在圈子里的内容只计入对应圈子未读，避免重复。
- 阅读对应个人动态或进入朋友主页并成功加载最新个人内容后，服务端更新该朋友的阅读位置。
- 好友申请、共同邀请等待办也可以点亮首页“朋友”入口，并在朋友页面顶部明确显示待处理事项。

#### 朋友关系展示

- 完整朋友列表默认不向其他用户公开，只允许用户本人查看。
- 访问他人主页时最多显示双方的共同朋友，不提供展开查看对方全部朋友的入口。
- 可以显示双方成为朋友的时间、邀请来源、共同圈子和共同经历，但每一项仍需通过原始权限检查。
- 朋友主页可以提供邀请对方加入小圈子的操作。
- 不提供私聊按钮，圆个圈不承担即时通讯功能。

#### 同圈非好友

- 同处一个小圈子但没有直接朋友关系的成员，可以看到彼此的头像、昵称和个人介绍，以满足圈内识别和协作需要。
- 双方可以查看共同圈子中原本有权访问的内容。
- 不能因此查看对方面向朋友发布的个人动态，也不能查看对方完整朋友列表或其他无关圈子。
- 没有朋友关系、共同圈子或其他合法内容关系的成员，默认不能浏览彼此的个人主页。

### 个人菜单、主页与设置

顶部头像是全局个人菜单入口。手机端点击头像后，从页面左侧滑出个人抽屉；桌面和平板宽屏使用头像下方的紧凑菜单。

#### 手机个人抽屉

- 手机一级页面的头像位于左上角，与从左侧出现的抽屉保持空间方向一致；桌面端仍可把头像放在右侧导航区。
- 抽屉宽度约为屏幕的 82% 至 86%，设置合理最大宽度，右侧保留半透明遮罩和明确的当前页面上下文。
- 顶部显示头像、昵称和“查看我的主页”，点击后进入自己的个人主页。
- 下方依次提供完整导航、存在草稿时才显示的“未完成的记录”、个人设置、通知、反馈、管理员入口和退出登录。
- 管理员入口只对管理员显示；没有草稿时不显示草稿空入口。
- 点击遮罩、关闭按钮或在抽屉内部向左滑动可以收起；打开抽屉时地址不变，进入具体页面后才切换路由。
- 首页、完整动态页、圈子列表和朋友列表可以把从页面左侧非交互空白区域开始的右滑作为辅助打开方式。
- 手势起点避开屏幕最边缘的系统返回区域；横向移动明显大于纵向移动并超过约 80 像素后才触发。
- 从按钮、链接、照片、横向内容组、输入框或其他交互元素开始的手势不打开抽屉。
- 地图、照片浏览器、足迹双视图、编辑器和其他依赖横向手势的页面完全禁用右滑打开抽屉。
- 点击头像始终是可靠的主要入口，滑动只作为辅助操作，不能成为发现菜单的前提。
- 打开时锁定背景页面滚动并正确管理键盘焦点；系统返回操作应先关闭抽屉，而不是直接离开当前页面。
- 抽屉使用连续暖色背景和清楚分组，不把每个菜单项做成独立卡片。

#### 我的主页与编辑资料

- 我的主页与朋友访问的个人主页复用同一结构，本人额外看到编辑资料、草稿、仅自己可见内容和归档入口。
- “编辑资料”只处理头像、昵称、简介、个人封面和封面焦点，不混入隐私、密码或通知设置。

#### 个人设置

- **隐私**：是否显示“最近来过”、个人动态默认可见范围和位置默认展示级别。
- **通知**：查看和管理重要朋友、重要圈子，以及后续加入的邮件通知类别。
- **账号安全**：修改密码、查看登录设备和退出其他设备。
- **数据与反馈**：查看数据处理说明并进入反馈入口。
- 第一版登录邮箱只读，不允许用户自行修改；后续增加重新验证身份、新邮箱验证、唯一性检查和会话安全处理后再开放。
- 第一版暂不提供删除账号功能。

## 核心内容类型

### 普通动态

用于记录当下感受或日常片段。

- 纯文字动态。
- 单张或多张图片。
- 可选心情状态。
- 可标记相关朋友。
- 支持评论和温和回应。
- 可发布到个人空间或小圈子。

### 游玩记录

用于保存已经发生的共同事件，而不是提前安排活动。

- 类型：旅行、聚餐、散步、看电影、探店或自定义类型。
- 实际发生日期，与帖子创建日期分开保存。
- 地点。
- 参与者。
- 所属小圈子。
- 文字与多张照片。
- 是否加入朋友地图，默认勾选，但用户可以在发布前关闭。
- 地点展示级别：精确地点、大致区域或不在地图展示。
- 支持圈内成员持续补充各自的照片和视角。

游玩记录中的“参与者”和“可见者”必须分开。例如，四人圈子中只有三人参加聚餐时，可以标记三位参与者，但仍由四位成员查看这段圈内记录。

### 共同相册

每个小圈子第一版只提供一个内置共同相册，不允许另外创建、命名或删除多个相册。共同相册是圈内照片的整理视图，不单独复制一份脱离动态的图片数据。每张照片都必须有明确上传者和来源内容。

- 圈内成员可以把同一小圈子动态或游玩记录中的照片导入共同相册。
- 相册不会自动收录所有圈内照片，由活跃成员主动选择需要整理进相册的内容。
- 导入只建立相册与原照片的引用关系，不复制图片文件，也不扩大原内容的可见范围。
- 相册条目显示照片上传者、来源动态的当前原文片段，并可以返回原动态详情；原文片段不使用大模型生成。
- 原动态文字被修改后，相册中的来源原文片段同步显示最新内容。
- 原照片被移除或原动态被删除后，对应相册条目自动消失，不保留孤立副本。
- 用户也可以从共同相册直接上传照片；系统会自动在该圈子中创建一条图片动态，并将照片加入相册。
- 相册直传生成的图片动态记录真实上传者，默认采用`圈内共同管理`，同时出现在圈子最近动态和共同相册中。
- 相册右上角的添加入口提供“从圈内动态选择”和“从手机上传”两个选项，并支持批量选择。
- 活跃成员可以把照片移出共同相册；该操作只删除整理关系，不删除仍然存在的来源动态和原照片。
- 已退出成员只能查看自己仍有历史访问权的相册内容，不能继续添加、移出或整理照片。
- 按时间、地点、人物和小圈子浏览，并可查看某一天或某次游玩的全部照片。
- 标记照片中的朋友时需要尊重对方确认和隐私设置。

#### 相册视觉布局

- 共同相册默认采用经过计算的等高自然拼贴：照片保持原始宽高比，同一行高度接近，行末根据可用宽度自然对齐。
- 不使用完全随机、高低失控的纯瀑布流，也不把所有照片强制裁成相同正方形。
- 照片间距保持细小且统一，不为每张照片增加独立卡片背景、厚边框或明显阴影。
- 手机端减少每行照片数量，避免横向照片被压得过小；平板和桌面端随宽度增加每行容量。
- 按时间形成克制的日期分组，让浏览顺序清楚，但日期标题不能频繁打断照片连续感。
- 图片尺寸尚未加载完成前使用已知宽高比预留位置，避免照片加载时整个相册跳动。

### 时间胶囊

时间胶囊用于把内容留给未来，而不是立即获得回应。

- 写给未来的自己。
- 写给某位朋友。
- 小圈子共同创建。
- 所有类型都需要标题和开启日期；简介可选，用于说明主题或留下开启前可见的话。
- 第一版支持文字和照片；语音与视频在媒体处理和存储能力完善后再增加。
- 写给自己或朋友的胶囊不设置单独关闭日期，创建者完成内容并主动封存后直接等待开启。
- 小圈子胶囊独有关闭日期，关闭日期必须早于开启日期；关闭前成员可以分别写入和修改自己的内容，彼此不可见。
- 到达小圈子胶囊关闭日期后停止新增和修改，已经提交的内容进入统一封存等待阶段。
- 封存后不能修改正文，只能等待开启。
- 到期后成为个人或圈子的长期回忆。
- 小圈子胶囊在开启前只显示标题、简介、关闭日期、开启日期、参与人数和已提交人数，不显示谁写了什么或谁写了多少，也不建立催促或排名机制。
- 有人到开启日期仍未写入时，胶囊照常开启；写入始终是自愿行为。
- 成员在封存时被明确列为接收者，即使后来退出圈子的活跃关系，到期后仍可以打开该胶囊。
- 尚无其他人提交内容时，创建者可以取消共同胶囊。
- 一旦其他成员已经提交内容，胶囊成为共同内容，任何个人都不能单方面取消，只能等待开启或通过紧急安全流程处理。
- 开启后，胶囊内容继续沿用封存时确定的接收者权限，不因之后的朋友关系或圈子状态自动扩大可见范围。

### 朋友地图

朋友地图记录共同经历发生在哪里，不提供实时定位。

- 地图地点主要由游玩记录生成。
- 点击地点可以查看相关照片、故事、参与者和日期。
- 可以查看与某位朋友或某个小圈子的共同足迹。
- 游玩记录默认加入朋友地图，发布者可以在发布前取消勾选。
- 用户可以选择“精确地点”“大致区域”或“不在地图展示”；地图接口不能向前端返回高于所选级别的位置数据。
- 使用精确地点时显示用户主动确认的 POI 或地图选点；使用大致区域时只返回城市或区县级展示坐标，不能先返回精确坐标再由前端模糊。
- 照片 EXIF 中的 GPS 信息不作为地图位置来源，默认清除或隔离；地图只使用用户主动确认并保存的位置。
- 地图内容始终继承原记录的权限。

#### 圈子足迹页面

圈子中的“足迹”使用独立页面承载游玩记录和地图两种视图，两者共享年份、参与者、类型等筛选条件以及当前选中的记录。

- 默认进入“记录”视图，按实际发生时间而不是帖子创建时间排列游玩记录。
- 记录视图使用连续时间线和轻分隔，显示日期、类型、参与者、地点、原文片段和照片摘要。
- 顶部提供克制的年份、参与者和类型筛选；新增按钮复用统一发布器并自动预选当前圈子和游玩记录。
- 没有地点或设置为“不在地图展示”的记录仍出现在记录视图，但不会进入地图接口。
- 手机端地图接近全屏，点击 Marker 后从底部出现可拖动的信息面板，展示地点和当前用户有权查看的相关记录。
- 记录视图中的地点可以打开地图并定位；地图中的记录可以进入详情，返回后恢复地图中心、缩放和筛选状态。

手机底部使用一个只属于足迹页面的连体分段控件：

```text
[ 记录 | 地图 ]
```

- 控件外层是弧度较大的圆角矩形轨道，内部有一个约占一半宽度的选中滑块，在“记录”和“地图”两个位置之间移动；它不是两个带中间分隔线的独立按钮。
- 两个位置使用简短图标和文字标识。点击任一位置时，选中滑块与页面内容使用一致方向的克制水平过渡。
- 用户拖动控件时，选中滑块跟随手指移动，页面切换进度可以同步响应；松手后根据距离和速度吸附到最近位置。
- 轨道可以使用对比度稳定的半透明纯色背景和轻微模糊，但不使用装饰性渐变；选中滑块需要在地图与记录背景上都清晰可见。
- 控件放在底部安全区域上方，保持稳定宽高和足够触摸面积。
- 在记录视图中可以整页向左滑动进入地图。
- 地图区域的单指拖动始终用于平移地图，不被页面切换手势抢占。
- 从地图返回记录时点击或拖动底部滑块式分段控件；控件本身支持双向跟手切换。
- 控件不是全站底部导航，离开足迹页面后不再显示，也不能遮挡地图信息面板的主要操作。
- 系统开启“减少动态效果”时直接切换状态，不使用跟手位移动画。

平板横屏和桌面端优先使用记录列表与地图并排布局。点击列表记录时地图定位，点击地图标记时列表定位到对应记录；空间不足时退回与手机一致的双视图切换。

#### 第一版地图范围

- 支持高德地图 POI 搜索、使用当前位置和地图手动选点。
- 支持 Marker，点击后显示当前用户有权查看的游玩记录摘要并进入详情。
- 支持按朋友、小圈子和时间筛选。
- 相近地点可以在前端聚合，避免标记互相遮挡。
- 暂不实现路线规划、实时定位、轨迹、热力图、围栏和复杂空间分析。
- 第一版不要求 PostgreSQL 安装 PostGIS；当数据量增长并真正需要大范围空间查询时再评估 PostGIS 和 GiST 索引。

#### 地点数据模型

地点不重复嵌入每条游玩记录，而是使用独立地点表并通过 `place_id` 关联。建议的数据层次如下：

```text
places
- id
- provider                 // amap
- provider_poi_id          // 地图手动选点时可以为空
- source                   // poi/current_location/map_pin
- name, address
- province, city, district
- adcode
- longitude, latitude
- coordinate_system        // gcj02
- fingerprint              // 非 POI 地点的去重辅助
- created_at, updated_at

administrative_areas
- adcode
- name, level
- center_longitude, center_latitude
- coordinate_system

outing_record_locations
- outing_record_id
- place_id
- place_name_snapshot
- address_snapshot
- display_level            // exact/area/hidden
```

- 中国大陆高德地点统一保存为 GCJ-02，并明确记录坐标系，不能把 WGS84 数据当作 GCJ-02 保存。
- 浏览器定位、外部设备坐标等来源必须标明原坐标系，并在服务端规范化后保存。
- `place_name_snapshot` 和 `address_snapshot` 用于避免高德 POI 后续改名时，旧回忆的历史展示跟着变化。
- `places(provider, provider_poi_id)` 在 POI ID 非空时建立部分唯一索引。
- `fingerprint` 用于地图选点等没有稳定 POI ID 的地点去重，但不要求在第一版解决所有近似地点合并问题。
- 为 `places(adcode)`、`outing_record_locations(place_id)` 和可在地图展示的记录建立查询索引。

#### 地点搜索、保存与地图查询

地点能力由服务端适配层封装，页面和业务模块不能直接依赖高德原始响应结构。

- `GET /api/places/search` 使用 Route Handler。输入至少经过防抖、长度限制、登录校验和限流；先搜索本地 `places`，结果不足时再由服务端调用高德 Web Service。
- POI 搜索结果返回平台自己的候选地点 DTO，不直接透传高德完整响应。
- 保存或修改游玩记录优先使用 Server Action，并在服务端完成 Zod 校验、权限校验、POI 再确认、地点复用和数据库事务。
- 图片不通过 Server Action 正文上传，使用独立上传接口或对象存储签名地址，避免大文件进入表单请求。
- `GET /api/map/records` 使用 Route Handler，接收地图范围与朋友、圈子、时间等筛选条件，适合地图拖动或缩放后重复请求。
- 使用当前位置或地图选点时，服务端通过逆地理编码补齐地点名称、地址和行政区编码；不能直接信任客户端提交的名称、POI ID 或坐标系。

#### 地图权限与安全 DTO

地图权限必须由统一的数据访问层处理，不能分别散落在页面、Server Action 和 Route Handler 中。

```text
requireSession()
canViewContent(userId, contentId)
getVisibleMapRecords(userId, filters, bounds)
toAuthorizedMapDto(record, viewer)
```

地图查询顺序固定为：

```text
确认登录
→ 按朋友关系、圈子成员关系和历史权限筛选内容
→ 关联游玩记录与地点
→ 按位置展示级别裁剪坐标
→ 返回专用地图 DTO
```

- `exact` 返回用户主动确认的 GCJ-02 POI 或选点坐标。
- `area` 只返回行政区中心坐标和区域名称，不能包含原始精确经纬度。
- `hidden` 在地图查询中完全排除。
- 无权限地点不能返回后再由前端隐藏，也不能泄露标题、参与者、数量或精确坐标。
- 所有 Server Action 和 Route Handler 都必须独立进行会话与权限校验，前端隐藏控件不属于安全措施。

#### 高德地图客户端与密钥

- 地图组件是独立 Client Component，通过 `next/dynamic` 关闭服务端渲染，并在卸载时销毁地图实例。
- 使用高德 JS API 2.0 与 `@amap/amap-jsapi-loader` 在线加载，不能下载高德资源后打包进项目。
- 浏览器 JS API Key、JS API 安全密钥和服务端 Web Service Key 分开申请和管理。
- Web Service Key 只能保存在服务端环境变量中，不能使用 `NEXT_PUBLIC_` 前缀。
- 正式部署优先按照高德建议，通过服务器或 Nginx 代理保护 JS API 安全密钥，并为正式域名配置白名单。
- 地点搜索输入进行约 350 毫秒防抖，取消过期请求，缓存常用结果并限制单次返回数量，以减少配额消耗。

#### 实现前的结构调整

1. 建立统一会话与内容权限数据访问层，地图、详情页和个人空间共享同一套判断。
2. 将后续增长的 Drizzle Schema 按认证、关系、内容、圈子和地点拆分，再统一导出。
3. 建立 `lib/server/places` 高德适配层和平台内部地点 DTO。
4. 建立环境变量校验，区分浏览器可见配置与服务端密钥。
5. 先完成真实内容、小圈子、历史权限和游玩记录，再接入地图页面，避免地图先于权限模型落地。

## 小圈子生命周期

### 创建与初始邀请

1. 发起者填写圈子名称并选择准备邀请的朋友。
2. 每位受邀者分别决定是否加入，接受前不能看到圈内内容。
3. 初始邀请自发出起 3 天内有效；到期仍未接受时自动失效。
4. 发起者与至少一位受邀者接受后，即至少形成两位成员，圈子可以正式启用。
5. 部分受邀者拒绝或邀请过期，不影响已经接受的成员正常建立和使用圈子。
6. 过期的初始受邀者不能在很久以后通过旧邀请直接加入；需要由当时的活跃成员重新发起新增成员流程。

### 新增成员

1. 任意活跃成员可以发起新增成员提案。
2. 发起者决定新人是否可以查看加入前的圈子历史，“允许查看历史内容”默认勾选。
3. 其他活跃成员在同意前必须看到该历史访问选项，避免在不知情的情况下扩大旧内容权限。
4. 所有活跃成员一致同意后，再由候选人本人接受邀请。
5. 候选人接受前不能查看圈内内容，也不属于活跃成员。
6. 勾选历史访问时，新成员加入后可以查看加入前的圈子内容；未勾选时，只能查看正式加入之后产生的内容。
7. 是否允许查看历史属于本次加入提案的一部分，审批过程中不能被发起者悄悄修改；如需改变，应重新确认。

### 活跃

- 可以发布动态和游玩记录。
- 可以评论、回应，并按内容管理方式参与编辑或删除。
- 可以发起新成员加入提案。

### 退出活跃关系

- 成员可以主动退出圈子的活跃关系，不需要解散整个圈子。
- 退出后不再接收或查看退出时刻之后新增的动态、照片、评论和编辑内容。
- 系统在退出时保存该成员的历史权限快照。
- 成员仍可只读查看退出前有权访问的历史记录。
- 退出后不能继续发布、评论、回应或管理内容。
- 该成员上传的文字、照片以及历史参与记录继续保留在共同档案中。
- 退出不会改变旧内容的创建者、参与者、创建时间或最后编辑信息。
- 退出成员不再参与新增成员、圈子设置和圈子解散等后续表决。

### 重新加入

已经退出的成员可以再次申请加入，同一个人第三次或更多次加入时继续使用相同流程，不设置固定次数上限。

1. 已退出成员发起重新加入申请。
2. 申请在 3 天内有效，过期后需要重新发起。
3. 当前所有活跃成员一致同意后，申请人才重新成为活跃成员。
4. 申请页面明确显示是否允许查看本次离开期间新增的内容；默认不允许，所有审批成员都能看到该选项。
5. 申请人原本已经获得的历史查看权继续保留，不因再次加入而丢失。
6. 重新加入后的新内容正常可见；离开期间的内容只有在本次申请明确获准时才能查看。
7. 每次重新加入都生成一条圈子系统动态；已经解散的圈子不能恢复活跃关系。

技术实现不能只在成员记录上反复覆盖一个状态。数据库应使用独立的成员周期表，为每次加入保存新的 `joined_at` 和 `left_at` 区间；当前周期的 `left_at` 为空。历史权限通过成员周期和当次重新加入审批中授予的离开期间访问范围共同计算。

### 个人封存

- 成员可以将圈子从自己的常用入口中隐藏。
- 不改变成员身份。
- 不影响其他成员使用。
- 可以随时重新显示。

### 解散

- 任意活跃成员可以发起解散申请，申请在 7 天内有效。
- 必须由所有活跃成员明确同意；未操作不视为同意，到期仍未获得全员同意时申请自动失效。
- 有成员拒绝时只显示“解散申请未通过”，不公开具体由谁拒绝。
- 圈子只剩一位活跃成员时，该成员本人确认即满足全体同意条件。
- 未能解散时，成员仍可以使用个人封存，从自己的常用入口隐藏圈子而不影响其他成员。
- 解散后停止新增、评论和编辑。
- 原有内容转为只读共同档案。
- 活跃成员可以查看解散时已有且原本有权访问的内容。
- 已退出成员仍只能查看其退出时历史权限快照内的内容，解散不会扩大其权限。
- 解散不等于删除，不能由某个人抹去整个圈子的内容。

## 编辑与最后修改

平台不把小圈子做成需要维护版本树的协作文档。正式内容只保留当前版本，并记录最少但有用的编辑信息：

- 最初创建者和创建时间。
- 最后编辑者和最后编辑时间。
- 当前内容管理方式，以及是否仍可由创建者升级为圈内共同管理。

交互与技术要求：

- 没有发生发布后修改时，不额外显示“最后编辑”。
- 修改后在卡片或详情中轻量显示“由眠眠最后编辑于……”。
- 不提供完整版本历史、字段差异或恢复旧版本功能。
- 修改日期、地点、参与者等关键信息时可以通知相关成员；普通文字修正不产生高频通知。
- 多人同时编辑时必须检测内容是否已变化，提示用户刷新或重新确认，避免后保存的人静默覆盖他人修改。
- 需要排查安全事件时，只记录必要的操作元数据，不保存供普通用户浏览的正文历史副本。

## 评论与回应

互动应温和，不建立热度竞争。

固定回应暂定为：

- 喜欢。
- 抱抱。
- 有同感。
- 下次一起。

动态卡片默认只显示一个中性的“回应”入口，用户点击后再展开具体选项，避免四个文字按钮长期占据内容区域。

- 卡片最多显示三位回应者的头像，超过后显示“等 N 人”。
- 回应总数不使用夸张字号、醒目颜色或排行榜进行强调。
- 用户可以进入详情查看谁作出了哪一种回应。
- 回应表达轻量态度，不代表参与活动，也不会改变内容管理权限。

同时支持：

- 普通评论。
- 回复某条评论。
- 为某张照片添加旁注。
- 查看参与回应的人，但不做热门排名。

评论与回复规则：

- 回复关系只展示一层，使用“眠眠 回复 小林”的形式表达，不进行无限视觉缩进。
- 评论者在仍拥有互动权限时，可以编辑或删除自己的评论；编辑后显示“已编辑”。
- 删除一条评论时，该评论下的所有回复一并删除，不保留“该评论已删除”等公开占位。
- 已有回复时，删除前必须明确提示将同时删除多少条回复，防止误操作。
- 删除完成后通知回复者等相关成员“你参与的一段评论已被删除”，通知不包含被删除的文字、图片或其他内容细节。
- 评论、回复和回应不会改变内容的管理方式或管理权限。
- 成员退出圈子的活跃关系后，历史内容整体只读，不能新增、编辑或删除自己以前的评论与回复。

平台不提供公开转发和传播链路。

## 动态内容与详情页

首页、圈子和个人空间中的每条动态在代码层可以复用同一个“动态卡片”组件，但“卡片”只表示内容单元，不代表视觉上必须有独立底色、边框和阴影。动态页是主要阅读场景，需要查看更多内容时进入独立详情页面，不在原信息流中无限展开复杂区域。

### 动态卡片

- 动态流与页面使用连续统一的背景，不使用一条动态一个白框的明显分割方式。
- 动态之间主要通过头像与作者信息的起点、舒适的上下留白以及很淡的细分隔线建立节奏。
- 默认不使用整块外边框和悬浮阴影；回应区、评论输入、引用内容等确实需要边界的局部区域才使用轻微底色或细边框。
- 显示作者头像、昵称、发布时间、发布空间和可见范围。
- 圈子内容明确显示来源圈子；点击圈子名称进入圈子详情。
- 点击作者头像或昵称进入对应个人主页。
- 直接显示必要且足够阅读的正文、照片、游玩摘要、回应、最多 10 条评论和最后编辑状态，不把正常阅读强迫到详情页完成。
- 评论不足 10 条时全部显示；超过 10 条时显示最近 10 条，并在动态中按从旧到新的顺序排列。
- 超过 10 条评论时显示“查看全部 N 条评论”，点击后进入独立详情页并定位到评论区域，不在信息流中无限展开。
- 较长正文在动态中先显示合理摘要，点击“展开”后在原位置显示完整正文，并提供“收起”；阅读正文不需要进入详情页。
- 正文展开与收起保持轻量，不使用夸张动画；展开后保持当前阅读位置稳定，收起时避免把页面突然跳到意外位置。
- 少量照片尽量完整展示；多张照片使用尺寸稳定的网格，并在最后一个位置显示剩余数量。
- 点击照片进入照片浏览器，不在信息流中直接展开成不受控制的长页面。
- 正文区域不承担进入详情页的操作，避免与选择文字、展开和收起发生冲突。
- 通过动态底部低调的“详情”入口、评论数量或“查看全部 N 条评论”进入独立内容详情页；通知、相册、地图和搜索结果也可以直接进入详情。
- 圈子内容聚合到个人空间时继续显示原圈子来源，不能伪装成该用户的个人动态。

### 内容详情页

- 详情使用独立页面路由，不覆盖在信息流上，也不要求通过复杂转场动画维持上下文。
- 展示完整正文、全部照片以及游玩日期、地点、参与者等结构化信息。
- 展示创建者、创建时间、最后编辑者和最后编辑时间。
- 回应、评论和回复集中放在正文之后。
- 从“查看全部 N 条评论”进入时自动定位到评论区域，并展示全部评论与回复。
- 根据当前用户身份和内容状态显示编辑、删除或更多操作。
- 从首页、圈子、个人主页或通知进入详情时，返回操作应回到原来的来源页面。
- 返回来源页面时恢复原筛选条件和滚动位置；可以使用短暂、克制的页面过渡，但功能不能依赖动画。

## 首页与自然浏览流

登录后的首页不是宣传落地页，而是朋友进入平台后的日常入口。

首页建议包含：

- 根据时间变化的简短问候。
- 快速发布入口。
- 圈子和朋友等常用入口。
- 从自己的个人动态和有权查看的圈子内容中受控随机选择的照片堆叠。
- 从人工文案库受控随机选择的暖心小纸条。
- 仅在有内容时出现的“那年今天”、地图故事和时间胶囊区域。
- 最新 5 条朋友、圈子和自己的动态。
- 进入独立完整动态流的“查看全部动态”入口。

首页文字应来自真实用户和真实内容，避免大段宣传式文案。

右侧或首屏的照片堆叠不应只是装饰：

- 点击照片后移动到最上层并轻微放大。
- 使用独立、明确的打开操作进入照片详情，避免“选中卡片”和“打开详情”含义冲突。
- 支持查看时间、地点、参与者和留言。
- 文字便签可以点击展开完整故事。
- 随机照片和暖心小纸条可以轮换，但同一用户在一段时间内保持稳定，避免每次刷新都变化。
- 交互动效保持克制、柔和，并支持键盘与触摸操作。

## 手机、平板与桌面端

平台应优先设计手机和平板体验，因为查看动态和上传照片通常发生在移动设备上。

布局根据可用宽度响应，不根据设备名称或 User-Agent 硬编码：手机小于 768 像素，平板为 768 至 1199 像素，桌面为 1200 像素及以上。

### 手机端

- 页面左右基础留白约 16 像素，正文不小于 16 像素，主要点击区域至少为 44 × 44 像素。
- 顶部栏约 56 像素，并适配刘海屏等安全区域。
- 采用自然的纵向内容流，不设置 App 式底部导航。
- 一级页面顶部保留左侧个人头像、中间站点标识以及右侧搜索和通知；二级页面优先保证返回和当前内容操作。
- 通过头像、圈子名称、地点、照片和“查看更多”自然进入二级页面。
- 二级页面提供明确的返回按钮和当前位置。
- 发布入口放在内容流顶部或当前圈子的就近位置，不使用持续遮挡内容的全站悬浮按钮。
- 全局菜单始终可找到动态、圈子、朋友、足迹、胶囊和设置。
- 编辑器使用全屏页面，减少软键盘挤压弹窗的问题。
- 动态流与页面背景保持连续，只用轻分隔线和适当留白形成节奏；移动端尤其不能把每条内容做成明显浮在另一层背景上的独立卡片。
- 地图使用接近全屏的地图与可拖动底部详情面板。

### 平板端

- 竖屏保持舒展单列，横屏在空间足够时使用双栏。
- 首页可以左侧显示动态、右侧显示回忆和圈子近况。
- 地图可以左侧显示记录列表、右侧显示地图。
- 相册使用更舒展的多列布局，列表与详情可以并排展示。
- 编辑器根据可用宽度和软键盘状态选择宽面板或全屏。

### 桌面端

- 页面整体最大宽度约 1180 至 1240 像素。
- 动态正文主栏约 680 至 720 像素，辅助栏约 320 像素。
- 使用顶部导航，不为了填满大屏无限拉宽正文或放大普通标题。
- 照片堆叠、回忆详情和圈内协作编辑可以利用更大的操作空间，但保持与移动端相同的操作含义。

### 个人封面

- 手机封面约为 16:7，平板约为 3:1，桌面约为 4:1。
- 桌面封面最大高度约 240 像素，不能占满首屏。
- 同一张封面照片根据容器比例裁切，用户可以设置焦点，系统优先保留人物面部或重要区域。
- 没有封面时使用个人主题色和克制的纸张质感，不使用装饰光球或夸张渐变。

### 可访问性与稳定性

- 键盘可以完成主要导航、发布、编辑和弹窗操作，当前焦点清晰可见。
- 颜色不能成为唯一状态提示。
- 支持系统“减少动态效果”设置。
- 图片提供合适的替代说明。
- 页面放大到 200% 时，文字、按钮和内容不能重叠或被裁掉。
- 横竖屏切换不能丢失草稿、已填写表单或当前浏览上下文。
- 实现后至少检查 360 × 800、390 × 844、768 × 1024、1024 × 1366 和 1440 × 900 等代表性视口。

## 照片上传与浏览

第一版只支持图片，视频和语音留到后续。每条个人动态、圈子动态或游玩记录最多包含 20 张照片。

- 调用手机相册或相机。
- 支持 JPEG、PNG、WebP 和苹果设备常见的 HEIC 等格式。
- 一次选择多张照片，但同一条内容不能超过 20 张。
- 上传前预览、排序和移除。
- 每张照片独立显示上传进度，失败后可以单独重试。
- 已上传照片跟随草稿保存在服务端，跨设备继续编辑时无需重新选择。
- 服务端保存高质量原图，同时生成适合信息流、详情页和缩略图使用的优化版本，避免移动端直接加载原图。
- 保持竖屏照片的自然比例。
- 默认清除或隔离 GPS 等精确定位信息；拍摄时间可以作为独立资料保存，但不自动向其他人公开。

### 动态与详情中的照片

- 单张照片尽量按自然比例展示；极端长图限制信息流高度，点击后查看完整图片。
- 两张照片左右排列，三张采用一大两小，四张采用 2 × 2，五至九张使用稳定网格。
- 超过九张时信息流只显示前九张，最后一格显示剩余数量；详情页展示全部照片。
- 发布者可以拖动调整照片顺序，第一张作为动态中的主要视觉照片。
- 信息流缩略图可以适度裁切以维持稳定布局，但全屏原图不能裁切；后续可以增加缩略图焦点调整。

### 全屏照片浏览器

- 使用纯净的深色背景，让照片成为视觉主体，不使用装饰性渐变或多余卡片。
- 支持左右滑动、双指缩放、拖动查看和当前序号，例如`3 / 18`。
- 照片说明、上传者和必要操作放在照片之外的深色留白区域，不用大块文字长期覆盖照片。
- 用户开始双指缩放、双击放大或拖动已放大的照片时，说明和操作栏快速、自然地淡出，照片本身不能因此移动或改变尺寸。
- 缩放操作结束后说明栏不强制立刻出现，用户轻点屏幕时再恢复；退出放大状态后也可以恢复基础操作栏。
- 淡入淡出时间保持短促克制；开启系统“减少动态效果”时直接切换显示状态。
- 可以查看照片说明、上传者和允许公开的拍摄时间，不展示设备型号或 EXIF GPS。
- 提供返回原内容的入口；有权限时可以移除照片，但危险操作不放在主要浏览手势附近。

圈子共同记录中的照片遵循以下删除规则：

- 每张照片始终记录真实上传者，并在照片详情和共同相册中显示。
- `仅我管理`的内容只能由内容创建者管理照片，其他成员不能向其中上传或移除照片。
- `圈内共同管理`的内容可以由任一活跃成员补充或移除照片，删除前应明确提示该照片会同时从动态和引用它的共同相册中消失。
- 移除照片不会改变内容的创建时间和管理方式；如仍有其他内容，动态继续保留。
- 成员退出圈子的活跃关系后，历史内容整体只读，不能再删除自己以前上传的照片。
- 隐私泄露、人身安全等紧急情况仍可以通过紧急隐藏和管理员审核处理。

## 个人主页

个人主页是一份有限可见的生活档案，而不是公开名片。

- 头像、昵称和简介。
- 当前状态。
- 个人动态。
- 与访问者共同参与的记录。
- 与访问者共同去过的地点。
- 双方共同所在的小圈子。
- 成为朋友的时间与邀请关系。
- 根据访问者身份展示不同范围的内容。

非朋友默认只能看到极少信息。没有朋友关系或共同圈子关系的平台成员不能通过昵称被搜索到。

## 通知中心

第一版只提供站内通知，不要求浏览器推送或邮件通知。通知中心使用独立页面路由，不长期设置“全部 / 互动 / 圈子”标签栏。

- 桌面和平板从全局主导航右侧的铃铛图标进入。
- 手机从首页、圈子等一级页面紧凑顶栏中的铃铛进入；内容详情等二级页面优先保留返回和当前内容操作，不强行塞入铃铛。
- 头像菜单中同时保留“通知”文字入口作为备用路径。
- 铃铛使用克制的未读圆点或数字，点击后进入通知页面；返回时恢复原页面位置。

通知页顶部优先显示尚未处理的邀请、加入、重新加入和解散申请。没有待处理事项时整个区域不出现；待办不会因为被阅读而消失，只在完成、拒绝或过期后离开。

普通通知按以下时间范围分组，并在每组内按时间从新到旧排列：

- 今天。
- 近 7 天，不含今天。
- 近 30 天，不含前两组。
- 更早。

通知项仍显示必要的具体日期或相对时间，分组只用于帮助浏览。右上角提供“全部标为已读”和按互动、圈子、系统类型筛选的菜单，不让筛选长期占据一排标签。

只发送有实际意义的通知：

- 朋友评论或回应了内容。
- 有人补充了共同记录。
- 有人修改了日期、地点或参与者。
- 有人邀请自己共同确认并邀请新人。
- 小圈子发起加入新成员的全员确认。
- 有人标记自己参与了一次活动。
- 时间胶囊即将开启或已经开启。

- 普通圈子的新动态默认只增加圈子的未读状态，不逐条进入通知中心。
- 用户可以把某个圈子标记为“重要圈子”，或把某位直接朋友标记为“重要朋友”；该标记只属于当前用户，对方和其他成员都不会知道。
- 重要圈子发布当前用户有权查看的新普通内容时发送站内通知。
- 重要朋友创建当前用户有权查看的新个人动态或圈子内容时发送站内通知；仅参与、评论或编辑别人内容不视为该朋友发布了新动态。
- 同一条内容同时命中重要朋友和重要圈子时只生成一条通知，并在文案中保留必要的作者和圈子来源。
- 重要标记默认关闭，只影响标记之后的新内容，不补发历史通知。
- 重要朋友可以从朋友主页更多菜单设置，重要圈子可以从圈子动态页更多菜单设置；设置项使用星标或开关表达，不放在公开资料中。
- 解除朋友关系、退出圈子、圈子解散或失去内容查看权限后，对应重要订阅停止生效；通知打开时仍需重新执行服务端权限校验。
- 圈子系统动态不因重要标记而逐条通知，仍只在圈内信息流中显示。
- 圈子名称、简介和封面等普通资料修改只显示为圈内系统动态，不额外通知所有成员。
- 同一条内容在短时间内收到的多个回应可以聚合为“某某等 N 人回应了你的动态”。
- 同一条共同记录的多次普通编辑应合并通知，避免形成打扰。
- 未读状态使用克制的圆点或数字，不使用持续闪烁、强烈红色或其他制造焦虑的方式。
- 点击通知进入对应内容后，返回操作应回到原来的通知列表位置。

后续可以增加用户主动开启的邮件通知，优先用于共同邀请、审批即将过期、时间胶囊开启等低频重要事件。邮件应按类别提供开关，只包含必要提示和站内链接，尽量不在邮件正文中暴露私人内容。浏览器推送和 PWA 通知同样留到后续评估。

## 反馈、空状态与错误提示

### 反馈

反馈入口放在头像菜单中，第一版将反馈保存在数据库并提供管理员查看页面。反馈类型包括功能建议、问题反馈、账号登录、隐私与数据和其他。

- 反馈状态第一版只需要“已收到”和“已处理”。
- 非匿名反馈可以在管理员回复后通过站内通知告知提交者。
- 截图由用户主动添加；系统不能自动附带动态正文、照片或精确位置。
- 可以附带经过裁剪的页面类型、浏览器类型和网站版本，不能直接保存含私人内容 ID 的完整页面地址。
- 隐私泄露和人身安全问题可以标记为需要尽快处理，但平台在没有正式值守能力时不能承诺全天候实时响应。

反馈必须由已登录成员提交，以阻止公开滥用，但用户可以选择匿名提交。

- 匿名提交时，服务端仍需短暂验证登录状态和执行限流，但反馈记录不保存 `user_id`、昵称、邮箱或会话标识。
- 请求日志不能主动记录匿名反馈的用户 ID、完整 Cookie、完整 IP 或反馈正文。
- 匿名只表示平台不建立反馈与账号的持久关联；用户自己填写的文字或截图仍可能暴露身份，界面需要明确提醒。
- 匿名反馈无法获得针对个人的回复，也不能在个人账号中查看处理进度。
- 非匿名是默认选项，用户主动开启“匿名提交”后才去除身份关联。

### 空状态

- 空状态使用一句自然说明和一个必要操作，不使用大段教程或夸张插画。
- 首页没有动态时可以显示“这里还很安静”和发布入口。
- 没有圈子、通知、地图记录或搜索结果时分别给出简短、符合当前页面的说明。
- 草稿为空时不显示草稿入口或空页面；只有存在草稿时才显示“还有 N 条未完成的记录”。

### 错误与恢复

- 表单输入错误显示在对应字段附近，不能只在页面顶部给出笼统错误。
- 网络失败时保留用户已填写内容；图片上传失败允许单张重试。
- 会话过期时先保留草稿，重新登录后返回原页面。
- 无权限页面不透露内容标题、作者、圈子、参与者或地点。
- 服务器错误不显示代码堆栈，只提供重试和反馈入口。
- 离线时显示轻量状态，并在连接恢复后继续同步草稿。
- 回应等乐观更新失败时恢复原状态并说明失败。
- 只有删除、退出、解散等重要或不可逆操作使用确认弹窗。

## 搜索

随着内容积累，平台需要让用户重新找到自己有权查看的内容和关系，但不能因此形成陌生人发现入口。

- 桌面端从顶部工具区的搜索图标进入；手机一级页面从顶栏右侧搜索图标进入独立搜索页。
- 搜索结果分为“内容”“人与圈子”“地点”三个栏目。
- **内容**第一版搜索有权查看的个人动态、圈子动态和游玩记录标题或正文，不搜索评论。
- **人与圈子**搜索直接朋友、当前或历史共同圈子成员，以及当前用户有权进入的圈子。
- **地点**搜索当前用户有权查看的游玩记录地点。
- 内容结果可以按年份、朋友、小圈子和内容类型筛选。
- 搜索结果使用连续背景和轻分隔，不做卡片墙或热度排名。
- 点击结果进入来源内容，返回后恢复搜索词、栏目、筛选和滚动位置。
- 没有搜索词时只显示当前浏览器中的最近搜索，不提供热门搜索或陌生人推荐。
- 最近搜索只保存在当前浏览器，可以单条删除或全部清空，不上传服务器。
- 输入请求使用防抖并取消已经过期的请求，避免快速输入时旧结果覆盖新结果。
- 所有搜索先在服务端形成当前用户有权访问的候选集合，再执行文字匹配和分页。

人物搜索必须先在服务端计算当前用户可发现的候选集合，再在候选集合中匹配昵称，不能搜索全部平台成员后交给前端过滤。候选集合包括：

- 当前用户的直接朋友。
- 与当前用户处于同一活跃小圈子的成员。
- 与当前用户共享圈子历史访问权的历史成员，但只能展示双方仍有权查看的关系信息。

没有任何上述关系的人不能通过昵称、头像或模糊关键词被发现。用于建立搜索关系的圈子不再对当前用户提供成员访问权时，也不能继续借此暴露对方。

游玩记录的参与者只能从该圈子的成员中选择，因此共同参与不额外建立一条人物发现关系，也不会绕过圈子权限。技术实现上通过朋友关系和圈子成员关系的服务端联合查询生成用户 ID 集合，再关联用户表搜索昵称。

第一阶段数据量较小时，中文内容搜索使用 PostgreSQL `ILIKE`和`pg_trgm`完成包含与模糊匹配，并为常用搜索字段建立合适索引。PostgreSQL 原生全文分词对中文能力有限；数据规模和搜索要求增长后，再评估中文分词插件或独立搜索服务，不在第一版提前引入额外基础设施。

## 权限规则摘要

| 场景 | 默认可见范围 | 可编辑与删除者 | 管理方式是否可切换 |
| --- | --- | --- | --- |
| 个人动态 | 朋友 | 发布者 | 不适用，始终仅自己管理 |
| 指定朋友动态 | 被指定的朋友 | 发布者 | 不适用，始终仅自己管理 |
| 仅我管理的小圈子动态 | 该圈子活跃成员 | 创建者 | 是，仅创建者可升级为共同管理 |
| 共同管理的小圈子动态 | 该圈子活跃成员 | 圈内活跃成员 | 否 |
| 仅我管理的小圈子游玩记录 | 该圈子活跃成员 | 创建者 | 是，仅创建者可升级为共同管理 |
| 共同管理的小圈子游玩记录 | 该圈子活跃成员 | 圈内活跃成员 | 否 |
| 已退出成员的圈子历史 | 退出时有权访问的内容 | 无，只读 | 不适用 |
| 个人时间胶囊 | 创建者，开启前不可读 | 封存前的创建者 | 不适用 |
| 共同时间胶囊 | 指定成员 | 封存前的参与成员 | 不适用 |
| 解散圈子的历史内容 | 原圈子成员 | 无，只读 | 不适用 |

所有后端接口都必须在服务器端再次验证权限，不能只依赖前端隐藏按钮。

## 设计基调

- 温暖、安静、亲密，但不幼稚。
- 保留当前奶油白、深绿、柔和粉色和浅黄色的配色方向。
- 文案像朋友之间的自然表达，避免广告口号和过度煽情。
- 动效用于表现照片、便签和回忆的层次，不制造炫技感。
- 页面应像不断生长的共同手账，但信息结构必须清楚。
- 优先保证触摸操作、照片浏览、文字可读性和隐私提示。

## 开发原则

- 身份、关系和权限优先于展示功能。
- 每个读取和写入接口都执行服务端权限校验。
- 图片原图与缩略图分开处理，避免移动端加载过重。
- 精确位置、联系方式等敏感信息默认不公开。
- 内容只保存当前版本和最后编辑信息；删除、权限变更、紧急隐藏等敏感操作保留不可篡改的审计元数据。
- 数据结构应允许未来迁移到独立服务器或其他云平台。
- 界面导航暂不锁死，根据真实内容和设备测试持续调整。

## 本地开发

需要 Node.js 22.13.0 或更高版本，以及 PostgreSQL 17。

首次安装 PostgreSQL 后启动服务并创建开发数据库：

```bash
brew services start postgresql@17
createdb friend_nest
```

从 `.env.example` 准备本机 `.env.local`，至少设置数据库地址、认证密钥、内部注册密钥和创始成员创建密钥。`.env.local` 不会提交到 Git。

安装依赖并应用数据库迁移：

```bash
pnpm install
pnpm run db:migrate
pnpm run dev
```

在当前这台开发电脑上，也可以直接运行已经准备好的启动脚本：

```bash
./scripts/start-local.sh
```

手机与电脑连接同一局域网时，可以通过电脑的局域网 IP 和端口 `3000` 访问。该地址还需要加入 `.env.local` 的 `TRUSTED_ORIGINS`，更换网络导致 IP 改变后应同步更新。

首次打开 `/setup`，使用 `BOOTSTRAP_KEY` 先创建管理员账号，再创建第二位普通创始成员。两人会自动成为朋友。创建完成后，普通新成员只能通过至少两位朋友确认生成的邀请码注册。

构建检查：

```bash
pnpm run build
```

准备上传 GitHub 前可运行：

```bash
pnpm run test
pnpm run build
git status --short
```

确认 `.env.local` 没有出现在 Git 状态中再提交。服务器部署时应在服务器上单独创建环境变量和 PostgreSQL 数据库，不能把本机 `.env.local` 或数据库文件上传到仓库。
