# 技术架构与部署

本文件记录运行架构、代码目录、本地开发和自托管部署边界。数据库细节见独立数据库文档。

## 实施状态

| 架构能力 | 状态 | 当前边界 |
| --- | --- | --- |
| Next.js、PostgreSQL、Drizzle、Better Auth 自托管应用 | `已实现` | 本地与标准 Node.js standalone 可运行 |
| 本地私有媒体与 Sharp 三规格处理 | `已实现` | 图片通过应用服务器上传和读取 |
| 持久圈子关系、退出档案与参与者权限 | `已实现` | `db/schema.ts`、`lib/circles.ts` 与圈子接口已接入 |
| 上海 COS 与数据万象直传处理 | `已确认设计` | 适配器、STS、CORS 与回调尚未实现 |
| 地图、评论、回应和完整通知架构 | `未来设想` | 不能按已有模块对外承诺 |

## 技术架构

- Next.js 16 + React 19 + TypeScript。
- PostgreSQL 17。
- Drizzle ORM 与 SQL 迁移文件。
- Better Auth 管理密码哈希、数据库会话和安全 Cookie。
- 标准 Node.js 自托管运行方式，生产构建支持 standalone 输出。
- 当前本地开发和单机测试使用 `PRIVATE_DATA_ROOT/media` 私有目录与 Sharp；启动脚本默认将 `PRIVATE_DATA_ROOT` 指向项目根目录下的 `.data`。文件不放进 `public/`，必须经过权限接口读取。
- 正式长期部署目标为香港应用服务器与 PostgreSQL、上海私有 COS 和数据万象。浏览器与 COS 直接传输图片本体，香港服务器只负责身份、权限、临时凭证、任务和元数据。
- 正式版本面向自租服务器上的标准 Next.js、Node.js、PostgreSQL 和对象存储部署。
## 项目结构

```text
along/
├── app/                    # Next.js 页面、布局与服务端 API
│   ├── api/                # 认证、注册、初始化和共同邀请接口
│   ├── components/         # 共享页眉外壳、分段控件和动态内容组件
│   ├── circles/            # 圈子列表、共同动态、成员周期和圈内发布
│   ├── compose/            # 手机端个人动态独立发布页
│   ├── drafts/             # 多草稿箱、筛选及手机端草稿编辑页
│   ├── feed/               # 当前用户有权查看的完整动态流
│   ├── friends/            # 直接朋友列表、搜索与私有备注
│   ├── home/               # 登录后的真实数据库主页与发布入口
│   ├── invites/            # 共同邀请管理页面
│   ├── notifications/      # 站内通知入口与后续通知列表
│   ├── posts/              # 手机端已发布动态独立编辑页
│   ├── profile/            # 本人及朋友的个人空间
│   └── setup/              # 首次部署初始化页面
├── db/                     # Drizzle 数据库连接与表结构
├── drizzle/                # 可提交、可追踪的 PostgreSQL SQL 迁移
├── lib/                    # 认证、关系查询以及媒体存储与处理适配层
├── public/                 # Logo 等浏览器直接访问的品牌静态资源
├── scripts/                # 本地启动与 standalone 构建辅助脚本
├── .env.example            # 环境变量模板，不含真实密码和密钥
├── next.config.ts          # Next.js standalone 自托管配置
└── package.json            # pnpm 脚本、依赖和当前版本号
```

当前代码边界：

- 账号、会话、朋友关系、邀请、个人资料、动态、服务端草稿及媒体元数据都连接 PostgreSQL，并通过 `drizzle/` 中的迁移维护。
- 建圈阶段只写入最长 24 小时有效的`circle_creation_requests（建圈请求）`和受邀者回应，不创建正式圈子。全部受邀者提前回应时立即结算；否则到期把未回应者视为未接受后结算。至少一人接受才原子创建活跃圈子和首批成员。
- 小圈子以`circle_member_relations（持久成员关系）`保存不会因退出重算的历史边界，以`circle_membership_periods（成员周期）`记录每次加入与退出，并用`active_period_id（当前活跃成员周期编号）`加速当前权限查询。`active_slot（当前活跃席位）`在数据库层限定每圈最多 10 位活跃成员；数据库在事务提交时校验活跃指针、席位、开放周期和圈子状态一致。
- 退出时生成一份只属于该用户的冻结档案，保存圈子名称、简介、正文与媒体关联；成员构成通过不可变成员周期按冻结时刻重建。重新加入删除旧档案并恢复当前内容，之后再次退出会生成新的当前档案。
- 最后一位活跃成员退出后，圈子进入带截止时间的冻结状态，所有未完成加入提案失效。最后退出者可在截止前恢复；内部维护任务在截止后硬删除圈子、内容、全体历史档案和失去引用的媒体。
- 圈子动态通过`post_participants（动态参与者）`表达共同经历与谁有关；个人主页只聚合主页主人仍是活跃成员且被明确列为参与者的原动态。
- 圈子内容写入与退出快照按圈子行锁串行执行；正文编辑另外锁定动态并使用客户端看到的最后编辑时间执行条件更新。后台媒体状态结算沿用同一圈子锁，但不更新正文版本时间。
- 本地图片位于 `PRIVATE_DATA_ROOT/media`；本地开发对应项目根目录下的 `.data`，不提交 Git。
- 媒体服务位于 `lib/media/`：本地存储与 Sharp 处理实现统一接口，数据库保存上传会话、处理任务和三种资源变体。正式 COS 适配必须采用对象到对象处理，不能为了复用本地 `Buffer` 接口让图片绕经香港服务器。
- 三种正式变体在生成时记录对象存储返回的 ETag；浏览器复用图片时由应用先检查权限，再用数据库中的 ETag 返回 304。未来 COS 适配通过上传结果、处理结果或不含图片正文的 HEAD 元数据取得版本标识，不把对象下载到香港服务器重新计算。
- 客户端媒体上传统一经过 `app/components/media-upload.ts`：重复选图按文件特征去重并追加，使用 `XMLHttpRequest.upload` 汇总真实字节进度；接口确认上传和任务创建后即可进入发布阶段，后续安全处理异步完成。
- 动态照片统一由 `PhotoViewer` 展示：桌面端使用有边界的居中舞台，移动端使用全屏手势层；媒体地址继续经过现有私有权限接口，查看器不会绕过服务端授权直接读取存储文件。缩略图与舞台之间使用独立固定几何层完成展开、收回和圆角插值，轨道视口遮住 GPU 合成产生的亚像素接缝；关闭完成与浏览器历史回退共用幂等收口，不能留下透明查看层或延迟页面下一次点击。
- 草稿通过独立的 `/api/drafts` 服务端接口按作者读写，列表支持个人、圈子、指定圈子筛选和分页；`draft_participants（草稿参与者）`保存圈子草稿的未发布参与者选择。首页快捷发布器刷新后从空内容开始，不再擅自挑选“最新一条”恢复；跨设备继续编辑统一从草稿箱进入。
- 草稿保存使用客户端看到的`updated_at（最后更新时间）`进行锁内乐观冲突校验。冲突时编辑器保留本地文字与图片，可以重新载入服务器版本或另存为新草稿；带图另存由存储适配层在服务器侧复制私有媒体对象，不能让浏览器下载后重新上传。未来 COS 适配必须改成同地域对象复制，不能让图片绕经香港应用服务器。
- 完整发布器在点击发布前先同步一份私有草稿，再用同一批媒体编号创建正式动态；权限变化或发布失败时草稿和已经上传的图片继续保留，重试不重复上传。发布事务成功后只删除本次草稿关系，正式媒体关联继续存在。
- 首页动态数据拆为独立的 React `Suspense` 区域，并用结构对应的 Skeleton 作为真实加载回退；服务端数据已准备好时不人为延迟。
- 圈子未读位置按成员期保存在 PostgreSQL，由用户实际进入圈子后通过受保护接口更新，首页服务端查询只返回聚合后的可见未读状态。
- `.env.local`、构建产物、依赖目录和本地数据库数据不会提交到 GitHub。
- 项目统一使用 pnpm，因此仓库只保留 `pnpm-lock.yaml`，避免不同包管理器的锁文件互相冲突。
## 开发原则

- 身份、关系和权限优先于展示功能。
- 每个读取和写入接口都执行服务端权限校验。
- 图片持久化为缩略图、查看器预览图和安全高清图；信息流不得加载高清图。
- 精确位置、联系方式等敏感信息默认不公开。
- 内容只保存当前版本和最后编辑信息；删除、权限变更、紧急隐藏等敏感操作保留不可篡改的审计元数据。
- 数据结构应允许未来迁移到独立服务器或其他云平台。
- 界面导航暂不锁死，根据真实内容和设备测试持续调整。
- 一级页眉统一由共享组件提供。桌面和平板采用“一体化头像 + 首页/圈子/朋友/个人分段导航 / 草稿箱 / 通知”，手机采用“头像与主要显示名 / 通知”，草稿箱从左滑个人抽屉进入；发布、草稿编辑和动态编辑属于手机全屏任务路由，由任务页自身提供返回栏并通过共享退出上下文执行未保存确认和离场动画，不与全局页眉叠加。页眉不放 Logo，圆形品牌图标由根布局配置为浏览器 favicon。朋友列表、草稿箱和邀请管理必须是独立路由。
- 用户资料将 Better Auth 兼容的 `name` 保留为主要显示名缓存，同时以独立的必填 `real_name` 和可选 `nickname` 表达真实业务含义。朋友备注按关系方向保存，圈子昵称按成员期保存，避免一个全局字段承担多种身份。

### 客户端加载分层

- 主导航、页面级手势、常用功能入口和必要的即时状态属于首屏客户端岛，必须随首屏水合并尽快可操作；静态欢迎文案、标题和普通链接尽量保留为服务端输出。
- 代码下载与解析、组件挂载、业务数据请求、图片加载分别管理。动态导入某个组件时，不自动预取它未来可能使用的全部数据或媒体。
- 默认收起但使用频率较高的模块，可以不进入首屏关键挂载路径，但应在页面稳定后的空闲时间预加载代码，并在 `pointerenter`、`touchstart`、`focus` 或相关内容进入视口时提高优先级。真正点击时如果仍未完成，先同步更新入口状态并展示轻量结构占位。
- 草稿箱卡片或个人、圈子发布入口可以按操作意图预加载完整发布器代码，但不能提前请求未打开草稿的媒体；照片查看器只预加载交互代码，预览图仍在用户打开相应照片时按权限请求。
- 低频且仅作者使用的编辑弹窗可以在点击后动态加载，但触发按钮必须立即反馈加载状态。不能为了统一形式把所有大型组件都改成点击后才加载。
- 性能验收同时关注首屏可点击时间和高频功能第一次操作延迟，并覆盖普通 4G、较慢大陆到香港网络和低性能手机；打包体积只是诊断指标，不是唯一目标。

## 本地开发

需要 Node.js 22.13.0 或更高版本，以及 PostgreSQL 17。

首次安装 PostgreSQL 后启动服务并创建开发数据库：

```bash
brew services start postgresql@17
createdb friend_nest
```

从 `.env.example` 准备本机 `.env.local`，至少设置数据库地址、认证密钥、内部注册密钥和创始成员创建密钥。`.env.local` 不会提交到 Git。

当前新图片目录位于 `PRIVATE_DATA_ROOT/media`。本地启动脚本固定使用项目根目录下的 `.data`；单机部署必须把 `PRIVATE_DATA_ROOT` 配置为构建目录之外的持久化磁盘绝对路径并纳入备份，不能依赖 Next.js 进程的当前工作目录。正式 COS 适配完成后，应用服务器不再长期保存图片。

安装依赖并应用数据库迁移：

```bash
pnpm install
pnpm run db:migrate
pnpm run dev
```

在当前这台开发电脑上，也可以直接运行已经准备好的启动脚本：

```bash
./scripts/start-local.sh
```

该脚本会启动 PostgreSQL 与固定在 `3000` 端口的 Next.js 开发服务器，并在服务器启动后自动打开 `http://localhost:3000`。本机界面开发显式使用 Webpack 热更新，优先避免组件与全局 CSS 偶发不同步；线上构建仍使用项目默认的完整构建流程。

手机与电脑连接同一局域网时，可以通过电脑的局域网 IP 和端口 `3000` 访问。该地址还需要加入 `.env.local` 的 `TRUSTED_ORIGINS`，更换网络导致 IP 改变后应同步更新。

使用 Cloudflare Quick Tunnel 在外网临时测试时，应用会自动信任 `https://*.trycloudflare.com`，不需要在每次生成随机域名后修改 `.env.local`。`TRUSTED_ORIGINS` 仍用于配置本机、局域网地址和以后购买的正式域名，多个地址使用英文逗号分隔。例如：

```env
TRUSTED_ORIGINS=http://localhost:3000,http://192.168.1.10:3000,https://along.example.com
```

Quick Tunnel 每次重新启动都可能生成不同域名，但 `*.trycloudflare.com` 通配规则会继续生效；应用本身重新启动后即可使用新的隧道地址。该规则只覆盖 Cloudflare 的 HTTPS 临时隧道子域名，不会信任任意网站。正式服务器应配置固定域名，并把 `BETTER_AUTH_URL` 和 `TRUSTED_ORIGINS` 都改为正式的 HTTPS 地址。

首次打开 `/setup`，使用 `BOOTSTRAP_KEY` 先创建管理员账号，再创建第二位普通创始成员。两人会自动成为朋友。创建完成后，普通新成员只能通过至少两位朋友确认生成的邀请码注册。

构建检查：

```bash
pnpm run test
pnpm run build
```

本地服务已经启动时，可以额外运行完整业务冒烟测试。测试会临时创建两位账号和朋友关系，验证登录、动态权限、三种图片规格、自动转正、EXIF 清除和资料修改，结束后自动清理：

```bash
pnpm run test:smoke
```

准备上传 GitHub 前可运行：

```bash
pnpm run test
pnpm run build
git status --short
```

确认 `.env.local` 没有出现在 Git 状态中再提交。服务器部署时应在服务器上单独创建环境变量和 PostgreSQL 数据库，不能把本机 `.env.local` 或数据库文件上传到仓库。

本地或单机服务器应由受控定时任务定期调用内部维护接口。它同时继续异常中断的媒体任务、回收过期 incoming 文件、结算到期建圈批次和加入提案，并硬删除已经超过冻结截止时间的圈子：

```bash
curl -X POST \
  -H "Authorization: Bearer $MEDIA_MAINTENANCE_SECRET" \
  https://along.example.com/api/internal/media-maintenance
```

建议每 5 分钟调用一次；业务权限以数据库截止时间为准，因此任务偶尔延迟不会延长恢复期限，但会延迟底层数据的实际清理。该接口不得暴露给浏览器，也不能使用登录 Cookie 代替独立运维密钥。完整媒体生命周期与正式 COS 方案见 [媒体上传与图片规格](features/media.md)，圈子冻结规则见 [小圈子与历史权限](features/circles.md)。
