# 地图与地点功能

本文件是地点保存、地图查询、位置模糊化和高德地图接入的权威说明。地图尚未进入当前版本实现。

## 实施状态

| 能力 | 状态 | 当前边界 |
| --- | --- | --- |
| 地点、游玩记录与地图数据模型 | `已确认设计` | 当前 Drizzle Schema 尚无对应表 |
| 高德 POI 搜索、当前位置和地图选点 | `已确认设计` | 尚无 Route Handler、适配器和客户端地图 |
| 精确、区域、隐藏三级位置权限 | `已确认设计` | 规则已确定，尚无运行实现 |
| 圈子记录/地图双视图与手势 | `已确认设计` | UX 已确定，页面尚未开发 |
| 路线、轨迹、实时定位和热力图 | `未来设想` | 第一版明确不实现 |

下文描述的是后续实现必须遵守的设计，不是当前系统已经可用的页面或接口。

## 朋友地图

朋友地图记录共同经历发生在哪里，不提供实时定位。

- 地图地点主要由游玩记录生成。
- 点击地点可以查看相关照片、故事、参与者和日期。
- 可以查看与某位朋友或某个小圈子的共同足迹。
- 游玩记录默认加入朋友地图，发布者可以在发布前取消勾选。
- 用户可以选择“精确地点”“大致区域”或“不在地图展示”；地图接口不能向前端返回高于所选级别的位置数据。
- 使用精确地点时显示用户主动确认的 POI 或地图选点；使用大致区域时只返回城市或区县级展示坐标，不能先返回精确坐标再由前端模糊。
- 照片 EXIF 中的 GPS 信息不作为地图位置来源，默认清除或隔离；地图只使用用户主动确认并保存的位置。
- 地图内容始终继承原记录的权限。

#### 圈子足迹页面

圈子中的“足迹”使用独立页面承载游玩记录和地图两种视图，两者共享年份、参与者、类型等筛选条件以及当前选中的记录。

- 默认进入“记录”视图，按实际发生时间而不是帖子创建时间排列游玩记录。
- 记录视图使用连续时间线和轻分隔，显示日期、类型、参与者、地点、原文片段和照片摘要。
- 顶部提供克制的年份、参与者和类型筛选；新增按钮复用统一发布器并自动预选当前圈子和游玩记录。
- 没有地点或设置为“不在地图展示”的记录仍出现在记录视图，但不会进入地图接口。
- 手机端地图接近全屏，点击 Marker 后从底部出现可拖动的信息面板，展示地点和当前用户有权查看的相关记录。
- 记录视图中的地点可以打开地图并定位；地图中的记录可以进入详情，返回后恢复地图中心、缩放和筛选状态。

手机底部使用一个只属于足迹页面的连体分段控件：

```text
[ 记录 | 地图 ]
```

- 控件外层是弧度较大的圆角矩形轨道，内部有一个约占一半宽度的选中滑块，在“记录”和“地图”两个位置之间移动；它不是两个带中间分隔线的独立按钮。
- 两个位置使用简短图标和文字标识。点击任一位置时，选中滑块与页面内容使用一致方向的克制水平过渡。
- 用户拖动控件时，选中滑块跟随手指移动，页面切换进度可以同步响应；松手后根据距离和速度吸附到最近位置。
- 轨道可以使用对比度稳定的半透明纯色背景和轻微模糊，但不使用装饰性渐变；选中滑块需要在地图与记录背景上都清晰可见。
- 控件放在底部安全区域上方，保持稳定宽高和足够触摸面积。
- 在记录视图中可以整页向左滑动进入地图。
- 地图区域的单指拖动始终用于平移地图，不被页面切换手势抢占。
- 从地图返回记录时点击或拖动底部滑块式分段控件；控件本身支持双向跟手切换。
- 控件不是全站底部导航，离开足迹页面后不再显示，也不能遮挡地图信息面板的主要操作。
- 系统开启“减少动态效果”时直接切换状态，不使用跟手位移动画。

平板横屏和桌面端优先使用记录列表与地图并排布局。点击列表记录时地图定位，点击地图标记时列表定位到对应记录；空间不足时退回与手机一致的双视图切换。

#### 第一版地图范围

- 支持高德地图 POI 搜索、使用当前位置和地图手动选点。
- 支持 Marker，点击后显示当前用户有权查看的游玩记录摘要并进入详情。
- 支持按朋友、小圈子和时间筛选。
- 相近地点可以在前端聚合，避免标记互相遮挡。
- 暂不实现路线规划、实时定位、轨迹、热力图、围栏和复杂空间分析。
- 第一版不要求 PostgreSQL 安装 PostGIS；当数据量增长并真正需要大范围空间查询时再评估 PostGIS 和 GiST 索引。

#### 地点数据模型

地点不重复嵌入每条游玩记录，而是使用独立地点表并通过 `place_id` 关联。建议的数据层次如下：

```text
places
- id
- provider                 // amap
- provider_poi_id          // 地图手动选点时可以为空
- source                   // poi/current_location/map_pin
- name, address
- province, city, district
- adcode
- longitude, latitude
- coordinate_system        // gcj02
- fingerprint              // 非 POI 地点的去重辅助
- created_at, updated_at

administrative_areas
- adcode
- name, level
- center_longitude, center_latitude
- coordinate_system

outing_record_locations
- outing_record_id
- place_id
- place_name_snapshot
- address_snapshot
- display_level            // exact/area/hidden
```

- 中国大陆高德地点统一保存为 GCJ-02，并明确记录坐标系，不能把 WGS84 数据当作 GCJ-02 保存。
- 浏览器定位、外部设备坐标等来源必须标明原坐标系，并在服务端规范化后保存。
- `place_name_snapshot` 和 `address_snapshot` 用于避免高德 POI 后续改名时，旧回忆的历史展示跟着变化。
- `places(provider, provider_poi_id)` 在 POI ID 非空时建立部分唯一索引。
- `fingerprint` 用于地图选点等没有稳定 POI ID 的地点去重，但不要求在第一版解决所有近似地点合并问题。
- 为 `places(adcode)`、`outing_record_locations(place_id)` 和可在地图展示的记录建立查询索引。

#### 地点搜索、保存与地图查询

地点能力由服务端适配层封装，页面和业务模块不能直接依赖高德原始响应结构。

- `GET /api/places/search` 使用 Route Handler。输入至少经过防抖、长度限制、登录校验和限流；先搜索本地 `places`，结果不足时再由服务端调用高德 Web Service。
- POI 搜索结果返回平台自己的候选地点 DTO，不直接透传高德完整响应。
- 保存或修改游玩记录优先使用 Server Action，并在服务端完成 Zod 校验、权限校验、POI 再确认、地点复用和数据库事务。
- 图片不通过 Server Action 正文上传，使用独立上传接口或对象存储签名地址，避免大文件进入表单请求。
- `GET /api/map/records` 使用 Route Handler，接收地图范围与朋友、圈子、时间等筛选条件，适合地图拖动或缩放后重复请求。
- 使用当前位置或地图选点时，服务端通过逆地理编码补齐地点名称、地址和行政区编码；不能直接信任客户端提交的名称、POI ID 或坐标系。

#### 地图权限与安全 DTO

地图权限必须由统一的数据访问层处理，不能分别散落在页面、Server Action 和 Route Handler 中。

```text
requireSession()
canViewContent(userId, contentId)
getVisibleMapRecords(userId, filters, bounds)
toAuthorizedMapDto(record, viewer)
```

地图查询顺序固定为：

```text
确认登录
→ 按朋友关系、圈子成员关系和历史权限筛选内容
→ 关联游玩记录与地点
→ 按位置展示级别裁剪坐标
→ 返回专用地图 DTO
```

- `exact` 返回用户主动确认的 GCJ-02 POI 或选点坐标。
- `area` 只返回行政区中心坐标和区域名称，不能包含原始精确经纬度。
- `hidden` 在地图查询中完全排除。
- 无权限地点不能返回后再由前端隐藏，也不能泄露标题、参与者、数量或精确坐标。
- 所有 Server Action 和 Route Handler 都必须独立进行会话与权限校验，前端隐藏控件不属于安全措施。

#### 高德地图客户端与密钥

- 地图组件是独立 Client Component，通过 `next/dynamic` 关闭服务端渲染，并在卸载时销毁地图实例。
- 使用高德 JS API 2.0 与 `@amap/amap-jsapi-loader` 在线加载，不能下载高德资源后打包进项目。
- 浏览器 JS API Key、JS API 安全密钥和服务端 Web Service Key 分开申请和管理。
- Web Service Key 只能保存在服务端环境变量中，不能使用 `NEXT_PUBLIC_` 前缀。
- 正式部署优先按照高德建议，通过服务器或 Nginx 代理保护 JS API 安全密钥，并为正式域名配置白名单。
- 地点搜索输入进行约 350 毫秒防抖，取消过期请求，缓存常用结果并限制单次返回数量，以减少配额消耗。

#### 实现前的结构调整

1. 建立统一会话与内容权限数据访问层，地图、详情页和个人空间共享同一套判断。
2. 将后续增长的 Drizzle Schema 按认证、关系、内容、圈子和地点拆分，再统一导出。
3. 建立 `lib/server/places` 高德适配层和平台内部地点 DTO。
4. 建立环境变量校验，区分浏览器可见配置与服务端密钥。
5. 先完成真实内容、小圈子、历史权限和游玩记录，再接入地图页面，避免地图先于权限模型落地。
